# Recommendation

## Verdict: keep-pm-and-incorporate — confidence: high

Keep pm as the conductor; do not switch to Comet. The debate's load-bearing fact survived scrutiny from both sides and my own repo check: pm and Comet occupy ADJACENT layers, not the same one. Comet is a single-change build ENGINE (bundles @fission-ai/openspec, wires Superpowers as script-gated phases, hard-blocks out-of-phase writes); pm is a portfolio CONDUCTOR that sits ABOVE any build lane (lane-agnostic epic index, LIFO detour stack, reconcile-on-resume, Honcho/tracker memory bridge). Confirmed by grep in both critiques: Comet has literally zero portfolio/detour/priority concept. Switching therefore forfeits pm's entire reason to exist to gain a build-phase enforcer — and it does so by adopting a 31k-line, 8-dependency, 0.4.0-beta.4 third-party runtime coupled to two fast-moving upstreams, which Rob would have to own or fork but did not write. That is a strictly worse trade for a single maintainer than keeping a 1,134-line zero-dep file he fully controls.

The pro-switch case has one genuinely strong core — the user's own global CLAUDE.md is built around gates that MUST fire, and pm is instruction-only — but it does not survive the pro-keep rebuttal I verified: pm already ships PreToolUse/SessionStart/PreCompact hooks, and Comet's own "hard" enforcement is itself nothing more than installed Claude Code hook scripts. So pm's lack of mechanical enforcement is a self-imposed philosophy ("the engine never mechanically blocks"), NOT the structural impossibility the synthesis claims. pm can add its own opt-in blocking guard hook without touching a network and without adopting Comet's runtime. Once that's true, the switch's headline argument — "only Comet can enforce" — is false: the enforcement gap is closeable on pm's own terms.

Critically, even in the pro-switch's own worst-case-for-pm world (the detour machinery is speculative — and I did confirm it is DORMANT here: empty detourStack, no detours.log file), the correct response is NOT Comet. Comet does zero portfolio management; if pm's coordination layer is over-built, the answer is to simplify pm toward a markdown backlog, not to adopt a tool that doesn't model portfolios at all. The switch verdict only wins under a narrow conjunction (see whenSwitchWouldBeRight). Absent that conjunction, keep-pm dominates.

Where the pro-switch and pro-keep critiques AGREE, and where I concur, is that the synthesis's "incorporate 8 things" half is where the real risk hides. Four of the borrows are stateful (events.jsonl audit log, ~/.pm cross-repo registry, machine-owned atomic detour CLI verbs, and above all the sha256 handoff-artifact GENERATOR) and push pm from "emit instructions" toward "generate and hash artifacts and do work" — i.e., a hand-built, solo, zero-dep reimplementation of the very Comet subsystems that keeping pm was meant to avoid depending on. Those must be reject-by-default, each gated against an explicit "does this still only emit instructions?" test. The verdict is keep-pm-and-incorporate — but with a deliberately SHORT, mostly-instructional incorporation list, plus two cheap safety fixes, not the sprawling eight.

## When switching to Comet WOULD be right

The switch-to-Comet verdict becomes correct only under a narrow, verifiable conjunction: (1) Rob's real workflow is genuinely single-change-at-a-time — the detour stack and reconcile gate proven dormant across ALL his repos over time, not just this dogfooding one (I confirmed they are dormant HERE: empty detourStack, no detours.log), making pm's portfolio apparatus speculative infrastructure that violates his own "resist complexity until it hurts" principle; AND (2) mechanical, unbypassable gate-enforcement is a hard requirement he will NOT meet via agent discipline or a self-authored guard hook (i.e., he wants the guardrail owned by tested upstream code, not maintained by himself); AND (3) he accepts depending on / forking a 31k-line beta runtime coupled to @fission-ai/openspec and obra/superpowers CLI contracts. If all three hold, the dominating move is Comet as the build+enforcement lane paired with a ~20-line markdown backlog replacing pm's conductor — enforcement pm lacks at a fraction of pm's maintenance surface. Separately and more modestly: Comet is worth a real future evaluation as a build LANE that pm ROUTES to (not a replacement) if Rob wants hard TDD/review gates — but that is NOT free, as the pro-keep critique correctly prices it: Comet stores state in .comet.yaml/.comet/run-state.json that pm's OpenSpec-implicit progress detection cannot read, so it requires building a new .comet-state adapter and stands up a duplicate OpenSpec install. That is a genuine project, a hybrid-adopt-both only if the enforcement is worth the second source of truth.

## Synthesis rationale (pre-debate)

**keep-pm-and-incorporate** — pm and Comet are not substitutes — Comet is a per-change build runtime that would sit BELOW pm as an alternative to the OpenSpec+Superpowers lane, while pm is the portfolio conductor above all lanes. Switching to Comet would forfeit pm's entire reason to exist (portfolio priority, detour stack, reconcile gate, Honcho/tracker awareness) to gain a build-phase enforcer, and would violate pm's zero-dependency instruction-layer law. Keep pm as the conductor. Selectively adopt a handful of Comet's proven disciplines that fit within pm's law: the sha256-anchored OpenSpec↔Superpowers handoff, the cross-repo project registry for /pm:upgrade, the bounded review-count table, and atomic/unforgeable state mutation for the highest-stakes detour-stack operations. Comet itself remains worth evaluating as a build LANE (a rigorous alternative to hand-run OpenSpec+Superpowers) that pm's conductor could route to — but that is a separate lane decision, not a conductor replacement.

**Risks flagged:**

- pm stays instruction-only, so its biggest gap vs Comet (no mechanical phase/gate enforcement) persists — a forgetful agent can still skip the reconcile gate. Mitigation is stronger re-injected reminders, not adopting Comet's runtime.
- pm's value is contingent on OpenSpec + Superpowers existing as companions; if the user's build discipline migrated wholesale to Comet, pm's OpenSpec-implicit progress/archive detection would need re-pointing at .comet.yaml/.comet state instead of openspec/changes.
- Incorporating Comet ideas risks scope creep that erodes pm's core virtue (one auditable zero-dep file). Each borrow must be gated against the architectural law: no npm deps, no network, no in-engine web server.
- Single-maintainer bus-factor risk applies to both; relying on Comet would additionally inherit its upstream coupling to @fission-ai/openspec + obra/superpowers CLI contracts.

## The adversarial debate

The synthesis recommended keep-pm-and-incorporate on the ground that Comet and pm are adjacent layers (a single-change build engine vs. a portfolio conductor), and proposed borrowing eight Comet disciplines within pm's zero-dep instruction-layer law.

The pro-SWITCH critique argued the synthesis's confidence was unwarranted: enforcement is what the user's own CLAUDE.md is built around (gates that MUST fire), and pm 'structurally cannot' provide it — the synthesis itself concedes a forgetful agent can skip the reconcile gate. It pressed three verified facts (a live later/blocked status bug, no atomic write, 85 tests that only check string output not agent obedience), noted several 'incorporate later' items are things Comet already ships (auto_transition), and called keep-and-incorporate a disguised multi-quarter solo rebuild of Comet. Its honest verdict was CONDITIONAL: switch the build+enforcement lane to Comet, keep pm's conductor only if a usage audit shows the detour machinery actually fires — the discriminating question the synthesis never asked.

The pro-KEEP critique agreed with keep-pm and strengthened it: grep confirms Comet has zero portfolio/detour concept; Comet is ~28x the code (31k lines, beta, 8 deps, bilingual docs already drifting) that Rob didn't write and can't fully audit; pm's loose filesystem coupling is resilience (survives upstream churn) not weakness; and pm's Honcho/tracker memory is Comet-absent. Its key correction: pm's non-enforcement is 'deliberately does not,' NOT 'structurally cannot' — pm already ships hooks and could add a blocking guard itself. It warned the real risk is the incorporation half, and that the stateful borrows (audit log, registry, atomic CLI verbs, sha256 handoff generator) must be reject-by-default.

My adjudication: the pro-keep critique wins the core because it dismantles the pro-switch's load-bearing 'only Comet can enforce' claim with a fact I verified, and because Comet is architecturally incapable of replacing pm's actual function. The pro-switch's fire-rate question is legitimate and I checked it — the detour machinery IS dormant in this repo — but that argues for simplifying pm, not for adopting a tool with no portfolio layer, and the user's global CLAUDE.md is strong evidence the workflow is intended discipline. Both adversaries converge with me on the one real weakness: the incorporation list must be short and mostly instructional, with every stateful borrow rejected by default.

### Pro-SWITCH critique (steel-man for adopting Comet)

_Position:_ pro-switch

**Strongest points**

- ENFORCEMENT IS THE PRODUCT, AND PM STRUCTURALLY CANNOT SHIP IT. The user's own global CLAUDE.md is built entirely around gates that MUST fire: two mandatory review gates, RED->GREEN TDD, the reconcile-on-resume gate, verification-before-completion, 'if untested it's not done.' Comet mechanically guarantees these via a transition-time evidence guard (recomputes from disk, exits 1 if unmet) PLUS a PreToolUse hook that blocks out-of-phase writes in real time. pm is 'instruction-only by architectural law' — the synthesis itself concedes 'a forgetful agent can still skip the reconcile gate.' For a user this obsessed with non-negotiable gates, a tool that ENFORCES them beats a tool that re-injects reminders and hopes. The gates pm cares most about are exactly the ones it can't guarantee.
- PM'S CENTRAL VALUE CLAIM IS ENTIRELY UNVALIDATED. pm's 85 tests (confirmed: 85) verify the ENGINE's string output — that conductor.mjs emits the right markdown. Not one of them tests whether an agent actually OBEYS the re-injected discipline. pm's whole reason to exist is 'make the agent follow the gates,' and there is zero evidence it does. Comet ships an actual mechanism for measured reliability (Pass@k capability vs Pass^k reliability, dimension-weighted rubrics, LLM-judge). No head-to-head exists, but the asymmetry is damning: Comet has a way to know its harness works; pm's core promise is asserted, never observed. For a user whose #1 principle is 'evidence-based partnership,' pm's headline claim is untested.
- PM'S HAND-MAINTAINED DIY APPROACH HAS ALREADY DRIFTED — LIVE, CONFIRMED BUG. README.md (lines 51-52) documents a status table with `later` ('deferred; excluded from NEXT UP') and `blocked` ('waiting on external dependency'); SKILL.md line 164, init.md line 28, and sync.md line 15 all instruct the agent to use them. But the engine's KNOWN_STATUSES = ['untriaged','queued','active','paused','planned','archived'] EXCLUDES both. So `/pm:epic add --status later` is rejected at conductor.mjs:804 with an error — a documented feature, part of the pitched roadmap-import onboarding flow, is broken. This is precisely the maintenance burden Comet amortizes across a real contributor base + CI + codecov: pm's docs and engine, maintained by one person, have silently diverged and shipped.
- THE ZERO-DEP INSTRUCTION-LAYER LAW IS A LIABILITY MASQUERADING AS A VIRTUE. It is the very constraint that forbids pm from EVER closing its #1 gap. The recommendation's own risk section admits it: pm 'stays instruction-only, so its biggest gap (no mechanical phase/gate enforcement) persists — mitigation is stronger re-injected reminders, not adopting Comet's runtime.' A tool whose founding architecture forbids it from solving its own worst problem should have that architecture questioned, not treated as sacred. Meanwhile pm has NO atomic write and NO concurrency guard (confirmed: no lock/rename/mtime in conductor.mjs) on state.json — its single system of record can be corrupted by a crash mid-write, and its highest-stakes operation (detour PUSH/POP) is done by hand-editing that file with no CLI safety net.
- SWITCHING KEEPS THE USER'S BUILD DISCIPLINE — IT DOESN'T ABANDON IT. Comet BUNDLES @fission-ai/openspec and wires in Superpowers disciplines (tdd_mode, review_mode) as script-gated hard constraints. The user already runs OpenSpec + Superpowers as their build lane; Comet is those same two systems wrapped in enforcement, routing (auto_transition, already shipped), a dashboard, and cross-device zero-context resume. The switch cost is far lower than the 'adjacent layers, no contest' framing implies: the user forfeits pm's portfolio/detour/Honcho bridge, NOT their spec-driven TDD discipline.
- 'KEEP-AND-INCORPORATE' IS A DISGUISED SOLO REBUILD OF COMET. The recommendation proposes borrowing EIGHT Comet subsystems — atomic writes, machine-owned unforgeable fields, an append-only audit log, a cross-repo registry, the sha256 handoff artifact, the bounded review-count table, recompute-don't-remember recovery, and an autonomy dial (which Comet already ships as auto_transition). That is a multi-quarter reimplementation of proven, tested infrastructure, by one maintainer, under a self-imposed zero-dep handcuff, during which pm stays strictly worse on every one of those dimensions. Rebuilding Comet by hand to avoid depending on Comet is the more expensive, higher-bus-factor path.

**Holes it found in the recommendation**

- THE ENTIRE 'PORTFOLIO CONDUCTOR' JUSTIFICATION RESTS ON AN UNAUDITED ASSUMPTION. The recommendation's load-bearing move is 'pm and Comet are adjacent layers, not substitutes — pm manages a portfolio of concurrent epics.' But it never establishes that the user actually runs many concurrent epics, or that the detour stack and reconcile gate have ever FIRED and saved something. For a single maintainer dogfooding one plugin, 'one active epic at a time' may be the norm — in which case Comet's multi-change discovery is sufficient and pm's portfolio apparatus is speculative infrastructure, violating the user's own 'Resist Complexity Until It Hurts' and 'Progress Is Features, Not Activity' principles. The recommendation asserts the machinery's value; it never verifies the machinery fires.
- THE ZERO-DEP LAW IS TREATED AS AXIOMATIC AND NEVER JUSTIFIED. 'Adopting Comet's runtime violates pm's instruction-layer law' is deployed as a self-evident dealbreaker, but the recommendation never argues why that self-imposed constraint should outrank mechanical enforcement of the exact gates the user's CLAUDE.md elaborately codifies. The law is circular: it's presented as the reason to keep pm, while simultaneously being the thing that guarantees pm can never match Comet's central capability. An honest recommendation would weigh the law as a cost, not treat it as a boundary condition.
- IT NEVER COSTS OUT THE INCORPORATION PATH. Eight borrowed subsystems are listed with S/M/L effort tags as if additive and cheap, but the recommendation waves away its own admitted risk ('incorporating Comet ideas risks scope creep that erodes pm's core virtue'). It never confronts that hand-rebuilding atomic writes, an audit log, a registry, and a sha256 handoff inside a 1134-line zero-dep file is a larger, riskier, more bus-factor-concentrated undertaking than adopting the tool that already ships and tests them.
- IT NEVER EVALUATES THE DOMINATING HYBRID IT ASSUMES AWAY. The recommendation presumes portfolio management REQUIRES pm's engine. It never considers: Comet for the build+enforcement lane (hard gates pm can't provide) + a 20-line markdown backlog + Comet's existing multi-change discovery + a thin Honcho hook. If the detour/reconcile machinery rarely fires, that hybrid delivers enforcement pm lacks at a fraction of pm's maintenance surface — and it's never weighed.
- IT DOWNPLAYS A REAL CORRECTNESS GAP WHILE FILING IT AS a 'nice-to-have.' The recommendation lists 'recompute, don't remember' as incorporation #7 (effort M, low priority). But Comet distrusts stored phase and re-derives from disk evidence, while pm TRUSTS stored epic status with no audit log and no atomic/unforgeable fields. That means pm can silently operate on stale or hand-forged state — a correctness and tamper-resistance gap for its highest-stakes operation, not a cosmetic polish item.
- BUS-FACTOR ASYMMETRY IS PRESENTED AS A WASH. Both are flagged 'single-maintainer,' but pm is a solo project with no external users, no CI-measured behavioral validation, and no eval evidence, while Comet has a contributor base, CI, codecov, published npm downloads, and a docs site. 'Full sovereignty' is scored as pure upside; the flip side — a solo, externally-unvalidated tool vs. a maintained, measured OSS project — is never priced in.

**Factual corrections**

- DIMENSION 3 OVERCLAIMS COMET'S PORTFOLIO ABSENCE. The synthesis says Comet has 'NONE... no cross-change index... grep for detour/portfolio/priority returns zero' and concludes 'No contest.' This is overstated. Comet's runtime has genuine multi-change awareness: discoverActiveChanges()/activeChanges(), and explicit disambiguation logic ('multiple active changes require an explicit change_id'; 'BLOCKED: multiple active changes require a current change'; the ambient resume probe refuses to auto-resume with >1 active change). Comet enumerates and selects among multiple concurrent changes. The word `detour` is genuinely absent — but what Comet actually lacks is PRIORITY ORDERING, an explicit LIFO detour STACK, and reconcile-on-resume, NOT the multi-item index the synthesis implies. pm's moat is real but narrower than 'no contest.'
- THE `later`/`blocked` STATUS ISSUE IS CONFIRMED REAL, NOT DISPUTED — but note it is CORRECTLY identified by the synthesis (incorporation #1), so it validates rather than corrects the synthesis. Flagging here only to prevent double-counting: it belongs in the pro-switch case as maintenance-burden evidence, and the synthesis's characterization ('--status later would be rejected by validation today') is accurate — verified at conductor.mjs:52 (KNOWN_STATUSES) vs README.md:51-52 / SKILL.md:164.
- THE 'AUTONOMY DIAL' (incorporation #8) IS NOT A NET-NEW IDEA TO BUILD — COMET ALREADY SHIPS IT. The synthesis frames it as 'analogous to Comet's auto_transition' as if aspirational; confirmed in domains/comet-classic/classic-state-command.ts, auto_transition is a live config field with precedence env COMET_AUTO_TRANSITION > project config > default 'true'. This strengthens the pro-switch read: several 'incorporate later' items are things Comet already provides today, so 'keep-and-incorporate' means rebuilding shipped Comet features by hand.
- VERSION PRECISION: Comet is at 0.4.0-beta.4 (package.json), not merely 'beta' generically — worth stating exactly since the recommendation leans on 'beta maturity' as a switch risk; the specific pre-1.0 point-release is accurate and the dependency count (8 runtime deps: @fission-ai/openspec, 5 @inquirer/*, commander, yaml) matches the synthesis exactly.

_Its honest verdict:_ The synthesis's 'keep-pm-and-incorporate, no contest' confidence is NOT warranted — but neither is 'abandon pm wholesale,' and I won't claim it, because the user's global CLAUDE.md elaborately codifies the four-role delivery system, minimal-vs-substantial detour classification, reconcileOnResume, and Honcho mirroring — that's real evidence these are load-bearing FOR THIS USER, and I cannot verify from the repos that they're ceremony. The honest maximal pro-switch verdict is CONDITIONAL and hinges on one empirical question the recommendation never asks: DO the detour stack and reconcile gate actually fire in practice? Move the build + enforcement lane to Comet now — it bundles the same OpenSpec+Superpowers pm orchestrates, adds hard evidence-guard + PreToolUse enforcement that pm's zero-dep law structurally forbids, ships auto_transition/multi-change resume/dashboard, and is externally maintained and tested rather than a solo, behaviorally-unvalidated engine with a live status-drift bug. Retain pm's portfolio engine ONLY IF a usage audit of .conductor/detours.log and the detour stack shows the machinery genuinely fires; if it rarely does, a 20-line markdown backlog + Comet's existing multi-change discovery + a thin Honcho hook dominates pm at a fraction of its maintenance surface. The verdict the synthesis should reach is: 'adopt Comet for build+enforcement; keep pm's conductor provisionally, pending a fire-rate audit' — not the frictionless 'adjacent layers, keep everything' it currently asserts. The single discriminating question is whether pm's coordination machinery earns its keep in observed use; the synthesis assumes yes and never checks.

### Pro-KEEP critique (steel-man for keeping pm)

_Position:_ pro-keep — keep pm as the conductor; do NOT switch to Comet. I agree with the synthesis's keep-pm verdict, and after verifying both codebases the case for keeping is even stronger than the synthesis states, while its "incorporate 8 things" half is where the real risk hides.

**Strongest points**

- Layer mismatch is decisive, and confirmed by grep: Comet has literally zero portfolio/detour/priority concept (`rg -i detour` = 0 hits across the codebase), while pm's entire reason to exist — lane-agnostic epic index, LIFO detourStack with {pausedEpic, reason, spawnedDetour, reconcileOnResume}, and a fresh-context reconcile gate — lives at a layer Comet doesn't model. Switching would delete pm's core competency to gain a single-change build engine. They are not substitutes.
- The fork-maintenance cost is quantitatively lopsided and I measured it: Comet is 31,373 lines of TypeScript/TSX across 12 domains + a React/Tailwind dashboard + a Python/pytest/Docker eval harness + a full bilingual (en/zh) skill tree that is visibly already drifting (the zh SKILL.md docs show mangled/interleaved tokens). pm is 1,134 lines of zero-dependency Node built-ins. That is a ~28x code-surface increase, in beta (0.4.0-beta.4), authored by a third party, with 8 runtime npm deps and an esbuild toolchain — a codebase Rob would have to fork and own but did not write and cannot fully audit.
- pm's 'loose coupling by convention' is resilience, not weakness. Because pm only reads filesystem conventions (tasks.md checkboxes, date-prefixed archive dirs) it survives OpenSpec/Superpowers version churn with no CLI contract to break. Comet hard-depends on @fission-ai/openspec ^1.5.0 and chases obra/superpowers skill contracts (already carrying compatibility hacks) — adopting it inherits a transitive dependency on two fast-moving upstreams. Zero-lock-in is a real, verified property of pm.
- pm keeps the Honcho memory bridge and external-tracker-awareness block (instruction-only, never contacts a tracker) — Comet has neither (grep confirms no memory/tracker concept). For a conductor whose job is to survive across sessions, repos, and tools, that cross-session memory story is load-bearing and Comet-absent.
- Both systems independently converged on 'filesystem-as-system-of-record + recompute on resume' for compaction survival, and pm's version is confirmed in its shipped hooks (SessionStart re-fires the brief on source=compact; PreCompact snapshots brief.txt). pm already solves the real failure mode with 3 small hooks and no runtime — buying Comet's 31k-line machine to get the same durability property is a bad trade.

**Holes it found in the recommendation**

- The recommendation frames hard phase-enforcement as a gap 'an instruction layer structurally cannot' close, but this is imprecise: pm ALREADY ships hooks (SessionStart/PreCompact/PostToolUse in hooks.json), and Comet's own 'hard' enforcement is just installed Claude Code hook scripts (comet-hook-guard.mjs) + guards that throw. A blocking PreToolUse gate is fully within pm's existing plugin model and would not violate its never-touch-a-network law. pm's non-enforcement is self-imposed philosophy, not a structural impossibility — so the recommendation's risk-mitigation line ('stronger re-injected reminders, not adopting Comet's runtime') under-states that pm could add a local guard hook itself, still without Comet, if it ever decided blocking was wanted.
- The 8-item incorporation list is where scope creep actually lives, and the recommendation under-weights its own warning. - 'Comet remains worth evaluating as a build LANE pm routes to' is treated as nearly free; it is not. Comet bundles its OWN OpenSpec (@fission-ai/openspec) and stores state in .comet.yaml/.comet/run-state.json — which pm's OpenSpec-implicit progress detection (it reads openspec/changes/*/tasks.md and date-prefixed archives) cannot see. - The bus-factor claim ('applies to both') is falsely balanced. pm's bus factor is trivially recoverable — 1,134 lines of zero-dep Node any competent dev owns in an afternoon. Symmetric framing hides an asymmetry that favors keeping pm.
