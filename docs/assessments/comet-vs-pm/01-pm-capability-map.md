# pm plugin — capability map (objective)

> `pm` v0.7.0 · MIT · Node 18+ · single zero-dependency `conductor.mjs` (~1,134 lines) · distributed via the Claude Code plugin marketplace. Mapped by an independent agent instructed to be candid about limitations (2026-07-13).

### pm-plugin

The `pm` plugin (v0.7.0) is a zero-dependency Claude Code plugin that adds a lane-agnostic project-management "conductor" layer above OpenSpec and Superpowers. Its entire runtime is one Node 18+ single-file engine (`scripts/conductor.mjs`, ~1134 lines) driven by slash-command docs, a skill, an agent, and three hooks. Architecturally it is deliberately an INSTRUCTION layer, never an INTEGRATION layer: the engine emits markdown/JSON instructions for the interactive agent to act on and never opens a network connection or calls an external tracker itself. It owns exactly three things the underlying build lanes don't — cross-epic priority ordering, an explicit LIFO detour stack, and epic links (including a reconcile relationship) — while delegating story-level progress to live-read `tasks.md`/plan checkboxes. `.conductor/state.json` is the system of record; `PROJECT.md` is fully generated. 85/85 unit tests pass (`node --test conductor.test.mjs`).

**Capabilities**

- **Lane-agnostic epic index with deterministic priority ordering** — Epics carry a lane (openspec|superpowers|claude-code|decision|external), priority (P0-P3/P?), status, and role. resolveEpics() merges stored metadata with on-disk OpenSpec changes and sorts by priority rank -> lane rank -> id, so ordering is stable and reproducible. Progress is resolved live by precedence (inline stories[] -> planPath checkboxes -> openspec tasks.md -> em-dash) rather than being re-tracked in state.json.
  - files: `plugins/pm/scripts/conductor.mjs`, `plugins/pm/skills/conductor/SKILL.md`
- **Detour stack + reconcile gate** — A substantial interruption PUSHes a frame {pausedEpic, reason, spawnedDetour, reconcileOnResume} onto state.detourStack (LIFO); the paused epic goes status=paused; the detour becomes its own lane-tagged P0 epic with resolves-blocker-for/may-invalidate links. On POP (/pm:resume) if reconcileOnResume was true, a fresh-context reconciler agent (agents/reconciler.md, sonnet, read-only) re-validates the paused proposal against what the detour actually shipped BEFORE any code is written. Minimal detours skip the stack and are just logged.
  - files: `plugins/pm/commands/detour.md`, `plugins/pm/commands/resume.md`, `plugins/pm/agents/reconciler.md`
- **state.json as system of record; PROJECT.md as generated view** — loadState/saveState read/write .conductor/state.json (pretty JSON). render() regenerates PROJECT.md entirely from state + live checkboxes and only rewrites when content (modulo the timestamp line) actually changed, avoiding mtime churn/spurious diffs. PROJECT.md carries a 'do not hand-edit' banner and embeds the exact briefing a fresh session would see.
  - files: `plugins/pm/scripts/conductor.mjs`
- **Compaction-surviving hooks (SessionStart / PreCompact / PostToolUse)** — SessionStart (matcher startup|resume|compact) runs `brief` which prints additionalContext JSON re-injecting the whole briefing — crucially it re-fires on source=compact so the index comes back the moment context is summarized away, and the brief re-embeds a compact RULES reminder. PreCompact (auto|manual) runs `snapshot` (render + write .conductor/brief.txt) because PreCompact cannot inject, only side-effect. PostToolUse(Bash) runs `commit-nudge`: on a detected `git commit` it logs DETOUR-COMMIT if inside a detour, self-heals an archived active pointer, re-renders, and nudges. All three no-op silently until /pm:init creates state.json (dormant-by-default, like openspec init).
  - files: `plugins/pm/hooks/hooks.json`, `plugins/pm/scripts/conductor.mjs`
- **Active-pointer verbs with single-active invariant** — set-active <id> / clear-active (0.7.0) are the CLI setters for the top-level .active pointer. activate() enforces that .active and status:active can never disagree: it sets the target active, demotes any other active epic to queued, and every path (set-active, update-epic --status active, add-epic --status active) keeps them in lockstep; moving the active epic off active clears the pointer. set-active rejects unknown/archived ids.
  - files: `plugins/pm/scripts/conductor.mjs`, `plugins/pm/commands/epic.md`
- **Epic hierarchy (single-parent tree)** — Optional parent field forms an acyclic single-parent tree, validated by a shared parentError() ancestor-walk (no self-parent, parent must exist, no cycle) reused by add-epic/update-epic/add-many. render() emits children indented under parents with a cycle guard and an X/Y children archived rollup; grouping is render-only so NEXT UP keeps global priority order (a P0 child of a P2 parent still surfaces at P0).
  - files: `plugins/pm/scripts/conductor.mjs`
- **Atomic bulk create (add-many) and write-back (update-epic)** — add-many --from <path|-> reads a JSON {parent?, epics[]} batch, validates everything up front (id format, uniqueness vs existing AND intra-batch, lane, status, parent refs + intra-batch cycles) and persists in a single write or writes nothing and exits non-zero — removing the write race of chaining add-epic. update-epic <id> mutates externalId/externalUrl/parent/status/priority on an existing epic under the same validation (the tracker-key write-back path).
  - files: `plugins/pm/scripts/conductor.mjs`, `plugins/pm/commands/epic.md`
- **External-tracker AWARENESS (instruction layer only)** — An optional tracker block {system, instance, projectKey, mechanism, statusIntent} recorded via set-tracker makes the conductor aware a repo mirrors epics to Jira/GitHub/Linear WITHOUT the engine ever contacting the tracker. It only shapes emitted instructions: writeRules() adds an 'External tracker sync' CLAUDE.md section assigning the agent ownership, and the brief's TRACKER SYNC line lists only honestly-computable drift (active-work epics lacking externalId). statusIntent maps lifecycle to a SEMANTIC target, never a literal transition; the agent resolves the real transition. Detection guidance explicitly refuses to infer a tracker from Git hosting.
  - files: `plugins/pm/commands/tracker.md`, `plugins/pm/scripts/conductor.mjs`
- **Versioning: migrations, upgrade staleness guard, changelog-on-upgrade** — pmVersion is stamped into state.json. MIGRATIONS is append-only/never-reorder, keyed by introducing release, applied SORTED by release so a repo several versions behind replays every missed migration in order (each additive/idempotent/back-compat). upgrade() refuses to run if a newer pm is installed than the running engine (forces /reload-plugins first) and prints the CHANGELOG delta for the versions it crossed. brief() nudges from the newest INSTALLED version so the prompt fires even before reload; changelog subcommand shows the delta on demand via zero-dep '## [x.y.z]' parsing.
  - files: `plugins/pm/scripts/conductor.mjs`, `plugins/pm/CHANGELOG.md`, `plugins/pm/commands/upgrade.md`
- **Honcho memory bridge (agent-driven, tool-agnostic)** — On every detour PUSH/POP the CLAUDE.md rules and command docs instruct the agent to also write a one-line memory via whatever Honcho MCP tool is connected ('paused X for Y' / 'resumed X, reconciled vs Y'), so the pivot relationship survives outside the repo. The engine itself does nothing here — it only emits the instruction.
  - files: `plugins/pm/scripts/conductor.mjs`, `plugins/pm/README.md`
- **Self-healing archive reconciliation** — isArchived() matches both an exact-id archive dir and OpenSpec's date-prefixed archive/<YYYY-MM-DD>-<id> form. reconcileArchived() (run in sync/commit-nudge/init/upgrade) stamps status=archived on archived epics and clears an active pointer aimed at one, so /pm:next advances without hand-editing state.json; render/brief also refuse to display an archived epic as active.
  - files: `plugins/pm/scripts/conductor.mjs`

**Strengths**

- Genuinely zero-dependency: the entire engine is one file using only node: built-ins (fs, os, path, child_process, url), enforced by CLAUDE.md as architectural law — trivial to audit, no supply-chain surface, no npm install.
- Clean architectural boundary held consistently: no code path opens a socket or calls a tracker; external sync is delegated to the agent as instructions. This is stated as law and actually true in the code.
- Strong compaction-resilience design: the same briefing is re-injected on SessionStart source=compact and snapshotted at PreCompact, directly targeting the real failure mode (losing the detour/reconcile thread when context is summarized).
- Defensive, back-compatible data handling: default-merge on load, validLink/normalizeLink guards, sorted append-only migrations, stamp-on-content-change render, cycle guards in both validation and rendering. A v0.4.1 state file loads unchanged.
- Well-tested for its size: 85 passing unit tests covering migrations (incl. multi-version replay), the single-active invariant, archive self-heal, tracker drift honesty, hierarchy rendering, and atomic add-many rollback. Tests are env-injectable (PM_CACHE_ROOT, CLAUDE_PLUGIN_ROOT) for fixture isolation.
- Dormant-by-default hooks: installed at user scope but no-op until /pm:init, so the plugin is safe to have installed globally without imposing itself on every repo — mirrors the openspec init model users already understand.
- Documentation is unusually rigorous and honest: the brief explicitly refuses to fabricate tracker transition drift it can't observe, and detection guidance repeatedly warns that Git hosting is not a tracker signal.

**Limitations**

- Documented statuses `later` and `blocked` are NOT in the engine's KNOWN_STATUSES (untriaged|queued|active|paused|planned|archived). README's status table, SKILL.md's status enum, and init.md ('set status active|queued|later') all reference them, but add-epic/update-epic/add-many VALIDATION would reject `--status later` or `--status blocked` — they can only enter via hand-editing state.json. A real doc/engine drift.
- Even if a `later`/`blocked` epic is hand-edited in, the lanes-rollup count only excludes `planned` (not later/blocked), so a deferred epic still inflates the lane counts — inconsistent with the 'planned is excluded from rollup' behavior.
- No CLI to remove/rename an epic, to PUSH/POP the detour stack, or to edit links: PUSH/POP, link creation, and reconcileNeeded clearing are all done by the agent hand-editing state.json (the docs say so explicitly). The 'never hand-edit' discipline applies to PROJECT.md and the active pointer, but detour-stack mutation is still manual JSON editing — the highest-stakes operation has the least CLI safety net.
- Enforcement of the reconcile gate, Honcho mirroring, and detour classification is entirely instructional (rules text + re-injected reminders), not mechanical. A non-compliant or forgetful agent can skip the reconcile gate; hooks are deterministic but 'blind to intent' by the plugin's own admission.
- commit-nudge detects commits by regex on the Bash tool_input command string (/git\s+commit/); commits made via a wrapper script, alias, or non-Bash path won't be logged as DETOUR-COMMIT, and the -m subject parser only handles simple quoted/unquoted forms.
- OpenSpec coupling is filesystem-implicit: progress and archive detection assume the openspec/changes and docs/superpowers/plans directory conventions. Non-OpenSpec/Superpowers lanes get no automatic progress source (render em-dash) unless inline stories[] are maintained by hand.
- No concurrency protection on state.json (read-modify-write with a final saveState); add-many is atomic within one invocation but two concurrent engine invocations could clobber each other. Low risk in single-session use, but not guarded.
- Version/upgrade UX is inherently multi-step and easy to get wrong: the staleness guard means the first upgrade INTO a version still runs the old engine until /reload-plugins, requiring users to internalize the update->reload->upgrade sequence per repo.

**Notable for pm**

- Scope is deliberately narrow and defended in writing: it is an instruction/orchestration layer over OpenSpec+Superpowers, explicitly NOT a second task tracker and NOT an integration layer. Positioning depends on those companions existing (OpenSpec is an optional npm CLI, not bundled; Superpowers recommended).
- The core value prop is a single failure mode: not losing the 'what were we doing / what did the detour invalidate' thread across a context compaction. Everything (hooks, re-injected rules, snapshot, reconcile gate) serves that one narrative — a tight, demoable story.
- Distribution is via the public cfdude-plugins Claude Code marketplace (/plugin marketplace add cfdude/cfdude-plugins; /plugin install pm@cfdude-plugins), then per-repo /pm:init. Node 18+ is the only requirement.
- It dogfoods itself: this repo is pm-conductor-managed, and CLAUDE.md encodes the release discipline (version bump + CHANGELOG + migration on schema change; tests green before commit; never --no-verify).
- Honcho integration is real but shallow-by-design — one-line memories on PUSH/POP, tool-agnostic. It is a bridge, not a dependency; the plugin functions fully without Honcho.
- The status/lane taxonomy and the doc-vs-engine drift on `later`/`blocked` is the most user-visible rough edge to fix next; it directly affects the roadmap-import onboarding flow the README pitches.
