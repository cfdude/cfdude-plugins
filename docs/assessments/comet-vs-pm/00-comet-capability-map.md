# Comet — capability map

> `@rpamis/comet` v0.4.0-beta.4 · MIT · Node ≥20 · TypeScript runtime + Python eval · bundles `@fission-ai/openspec`. Fork at `cfdude/comet`. Mapped by 4 parallel deep-reader agents (2026-07-13).

Comet is **"an Agent Skill Harness for turning ideas into evaluated workflows"** — a per-change, evidence-gated five-phase state machine (open → design → build → verify → archive) that manages exactly ONE change at a time per git branch, plus a skill-authoring platform, a scientific eval platform, and a browser dashboard.

### engine+state

Comet 0.4.0's engine is a per-change, evidence-gated five-phase state machine (open -> design -> build -> verify -> archive) that manages exactly ONE change at a time per git branch. It has NO portfolio, NO priority ordering across changes, and NO detour/park-resume stack. Each change is an isolated unit (its own branch or worktree) with its own .comet.yaml (user-visible projection) plus a machine-owned .comet/run-state.json (engine Run state) and an append-only .comet/state-events.jsonl audit log. Phase exits are enforced two ways: (1) a transition-time guard that recomputes evidence from disk before advancing the phase field, and (2) a PreToolUse write hook that blocks source-file edits that don't belong to the current phase. Recovery is deterministic and zero-context: any session/device recomputes the exact current step purely from on-disk state + file evidence, never from conversation history. Multiple changes can physically coexist, but the engine refuses to guess between them - it forces an explicit single current change selection (branch-scoped) or blocks the write.

**Capabilities**

- **Five-phase state machine with a single canonical transition table** — Phases are open -> design -> build -> verify -> archive. Nine events drive it: open-complete, design-complete, build-complete, verify-pass, verify-fail, archive-confirm, archive-reopen, archived, preset-escalate. applyClassicTransition() is the ONE authority for phase mutations, shared by the guard (comet-guard --apply), the CLI (comet-state transition), and the archive path. Workflow variants branch the flow: full runs all five phases; hotfix/tweak skip design (open-complete jumps straight to build). Every transition asserts current phase == expected 'from' phase and emits a typed effects list. verify-fail loops phase back build->verify->build (preserving evidence); archive-reopen rewinds archive->verify.
  - files: `domains/comet-classic/classic-transitions.ts`, `domains/comet-classic/classic-state.ts`
- **Per-change state model split into three files** — .comet.yaml (per change under openspec/changes/<name>/) holds user-visible fields: workflow, phase, build_mode (execution mode: subagent-driven-development | executing-plans | direct), verify_mode (light|full), verify_result (pending|pass|fail), verification_report path, archive_confirmation (pending|confirmed), archived, isolation (branch|worktree), auto_transition, plus handoff/design links. Engine Run state (run_id, skill, skill_hash, current_step, iteration, pending action, trajectory/context/artifacts/checkpoint refs, status, retries) lives separately in .comet/run-state.json, linked from yaml by only run_id. .comet/state-events.jsonl is an append-only audit of successful transitions. Old yaml with embedded Run fields auto-migrates on first read.
  - files: `domains/comet-classic/classic-state.ts`, `domains/engine/state.ts`, `assets/skills/comet/reference/comet-yaml-fields.md`
- **Phase-exit enforcement via evidence guards (the transition gate)** — Before a phase can advance, classic-guard.ts runs phase-specific evidence checks against the filesystem - open: proposal/tasks(/design for full) exist and match configured language; build: isolation selected, build_mode selected+allowed, subagent_dispatch confirmed if applicable, tdd_mode & review_mode selected (full only), tasks.md ALL boxes checked, plan tasks checked, build command passes; verify: tasks complete, verification command passes, verification_report exists, branch_status=handled. If any check fails the guard prints BLOCKED and exits 1 - the phase field is not advanced. The same hard constraints are duplicated in comet-state transition's require* functions. Machine-owned fields (archive_confirmation, classic_profile, classic_migration, run_id) cannot be forged via set; phase cannot be set directly except the COMET_FORCE_PHASE=1 repair hatch.
  - files: `domains/comet-classic/classic-guard.ts`, `domains/comet-classic/classic-state-command.ts`
- **Phase-exit enforcement via PreToolUse write hook (the 'agent can only do allowed work' gate)** — classic-hook-guard.ts is installed as a PreToolUse hook into each platform's settings; it inspects the target file_path of every write and returns exit 2 (BLOCKED) if the write is illegal for the current phase. open/design/archive phases block ALL source writes (only allow proposal/design/tasks/spec/state/handoff artifacts and whitelisted paths); build/verify allow source writes. Full workflow at build with an empty design_doc is blocked until a Design Doc is recorded. This constrains the agent to phase-appropriate work in real time, independent of the transition guard.
  - files: `domains/comet-classic/classic-hook-guard.ts`
- **Auto-transition routing (decoupled from phase advancement)** — auto_transition controls ONLY whether the next skill is auto-invoked after the guard advances a phase - it never gates the phase field update itself (guard --apply always advances phase). comet-state next <change> reads phase+workflow+auto_transition and emits a text protocol: NEXT: auto|manual|done, SKILL: <comet-open|design|build|hotfix|tweak|verify|archive>, HINT: (manual only). Three-layer precedence: env COMET_AUTO_TRANSITION > change-level .comet.yaml > project .comet/config.yaml > default true. Phase->skill mapping differs by workflow (build maps to comet-hotfix/comet-tweak/comet-build).
  - files: `domains/comet-classic/classic-state-command.ts`, `docs/operations/AUTO-TRANSITION.md`
- **Deterministic zero-context / cross-device recovery** — Recovery never reads conversation history - it recomputes state from disk. comet-state check <change> <phase> --recover prints structured recovery context (phase, done/pending fields, and an explicit 'Recovery action' string) for each phase. The deterministic resolver (resolveClassicStepId) recomputes the exact current step from ClassicState + file evidence (tasks checked, handoff present, report present, etc.), so any device/session lands on the same step ('engine-projection' runtimeMode). The engine layer persists run-state.json + trajectory.jsonl + checkpoint.json + context.md; the loop/resolver replay these to resume mid-run. Recovery principle is explicit: 'Do not infer phase from conversation history.'
  - files: `domains/comet-classic/classic-resolver.ts`, `domains/comet-classic/classic-diagnostics.ts`, `assets/skills/comet/reference/context-recovery.md`, `domains/engine/loop.ts`
- **Ambient resume probe (implicit re-entry without an explicit /comet command)** — comet-resume-probe probe --stdin decides whether to resume when the user speaks without invoking a /comet command. It discovers active changes under openspec/changes/, checks git dirty state, and returns one of auto_resume | ask_user | out_of_scope | none with evidence and confidence. Critically: with MORE THAN ONE active change it will NOT auto-resume - it returns ask_user 'multiple active changes require a change name' unless the utterance names one; dirty worktree also forces ask_user (attribution required).
  - files: `domains/comet-classic/classic-resume-probe.ts`
- **Engine core: deterministic loop, guardrails, and skill resolver** — The generic engine (used by the Skill platform, mirrored by Classic via a resolver) runs a decide->act->recordOutcome loop over a Run state. guardrails.ts authorizes each action: iteration budget (maxIterations), allowedSkills/allowedTools/allowedAgents allowlists, per-action retry budget, and confirmationRequiredFor (user-confirmation gating). loop.ts proposes exactly one pending action at a time (status running->waiting), records success/failure, increments iteration, and marks completed when the resolver returns no next step. Orchestration is either deterministic (resolver-driven) or adaptive (agent supplies the candidate action).
  - files: `domains/engine/loop.ts`, `domains/engine/guardrails.ts`, `domains/engine/resolver.ts`

**Strengths**

- Single source of truth for phase transitions: one applyClassicTransition() table is shared by the guard, the CLI transition command, and archive - no divergent copies of 'what advances a phase.' Constraints are deliberately duplicated in both the guard and the transition command so neither entry point can be bypassed.
- Evidence-based, not memory-based: current step is recomputed from on-disk artifacts + state, making recovery fully deterministic and identical across devices/sessions after context loss. Recovery explicitly forbids inferring phase from chat history.
- Two independent enforcement layers: a coarse transition gate (can you LEAVE this phase?) plus a fine-grained PreToolUse write hook (can you WRITE this file right now?). The hook actively constrains the agent to phase-appropriate work in real time, not just at phase boundaries.
- Clean separation of user-owned vs machine-owned state: .comet.yaml is a tidy user projection; engine Run state and the audit log are separate, and machine-owned fields (archive_confirmation, classic_profile, run_id, and phase itself) cannot be hand-forged, closing state-machine bypass holes.
- Atomic writes throughout (temp-file + rename) for yaml, run-state, and current-change selection, reducing corruption risk on interrupted writes.
- Zero-runtime-dependency posture (Node built-ins + a yaml parser), cross-platform (no bash/WSL), which lowers the barrier for the 'runs anywhere / resumes anywhere' story.

**Limitations**

- ONE change at a time, no portfolio. There is no priority ordering, ranking, queue, or 'what's next across changes' concept anywhere in the engine (grep for detour/portfolio/priority/queue returns zero hits). Multiple changes may physically coexist under openspec/changes/, but the engine treats them as mutually-exclusive candidates, not a managed backlog.
- No detour / park-and-resume stack. Unlike a PM conductor's detour stack, Comet has no mechanism to pause the active change, do an urgent thing, and pop back with a reconcile gate. The only in-flow rewinds (verify-fail, archive-reopen, preset-escalate) all mutate the SAME change in place; they are not a stack of suspended work.
- Concurrency is resolved by forcing a choice, not by managing it. With >1 active change: the write hook BLOCKS all source writes until you run 'comet state select <change>', and the resume probe refuses to auto-resume. The 'current change' is a single branch-scoped selection (.comet/current-change.json) that goes stale if you switch git branches - so concurrent changes are really 'one change per branch,' disambiguated manually.
- Cross-change relationships are invisible to the engine. Because each change is isolated to its own branch/worktree and there is no index spanning them, dependencies, ordering, or 'epic A must precede B' cannot be expressed - that coordination is entirely on the human/agent, outside Comet.
- The interruption model assumes branch isolation. If two changes share a worktree or an urgent fix lands on the same branch, the single current-change selection and phase-scoped write hook can produce blocked writes or mis-attributed edits (dirty-worktree forces manual attribution).

**Notable for pm**

- Direct contrast to a pm conductor: Comet deliberately stops at the single-change boundary. It owns the RIGOROUS build of one change (evidence gates, phase-scoped write enforcement, deterministic resume) but has no portfolio, no priority queue, and no detour stack. That is precisely the layer a pm plugin sits ABOVE - Comet is a 'one-lane build engine,' not a multi-epic conductor.
- The detour gap is real and structural, not an oversight to route around: Comet's answer to 'urgent interruption' is to create a separate change on a separate branch and rely on branch-scoped current-change selection to keep them apart - with no automatic push/pop, no reconcile-on-resume, and no priority. 'Detour stack + reconcile gate + priority ordering' is a differentiator a conductor has that Comet does not.
- Comet's strongest transferable idea is evidence-recomputed, zero-context recovery: the current step is always re-derived from files on disk, never from chat memory, so it survives compaction and moves across devices identically. Any conductor claiming cross-session durability should adopt this 'state of record is the filesystem, recompute don't remember' discipline.
- Two-tier enforcement (transition gate + PreToolUse write hook) is worth noting: Comet doesn't just check at phase boundaries, it actively prevents out-of-phase file writes as they happen - a hard guardrail an instruction-only conductor layer cannot itself provide.
- auto_transition cleanly separates 'advance the phase' (always happens when evidence passes) from 'auto-invoke the next step' (user-configurable) - a useful pattern for a conductor deciding how autonomous to be between gates.

### openspec+superpowers-integration

Comet is a "dual-star" workflow runtime that orchestrates OpenSpec (owns WHAT — proposal/spec/tasks lifecycle + archive) and Superpowers (owns HOW — brainstorming/design/plan/execution/review/verification) over a single resumable five-phase state machine (open → design → build → verify → archive) recorded per-change in `openspec/changes/<name>/.comet.yaml`. The integration lives at two distinct layers. (1) An INSTALL layer (`domains/integrations/openspec.ts`, `superpowers.ts`) that shells out to install the `@fission-ai/openspec` CLI globally and the `obra/superpowers` skill bundle across ~40 agent platforms — it does no runtime linking. (2) An ORCHESTRATION layer: a router skill (`assets/skills/comet/SKILL.md`) plus seven phase sub-skills (`comet-open/design/build/verify/archive` + `comet-hotfix`/`comet-tweak` presets) that emit `Skill`-tool load instructions to nested OpenSpec and Superpowers skills, gated by a deterministic Node runtime (`domains/comet-classic/*` compiled to `assets/skills/comet/scripts/comet-runtime.mjs`). Comet never edits the vendored OpenSpec/Superpowers skills; it wraps them, passes context via a hash-verified handoff package, and enforces ordering with a phase guard, a per-round anti-drift rule, and a PreToolUse hook.

**Capabilities**

- **Install-time integration of OpenSpec + Superpowers (multi-platform)** — openspec.ts runs `npm install -g @fission-ai/openspec@latest`, then `openspec init <target> --tools <ids> --profile custom` with a temp XDG_CONFIG_HOME + backup/restore of the user's real openspec config so ALL workflows (propose/explore/new/continue/apply/ff/sync/archive/bulk-archive/verify/onboard) are enabled; it also mirrors/migrates openspec's opencode-written skills+commands into opencode-compatible dirs (zcode) since openspec hardcodes the opencode skillsDir. superpowers.ts runs `npx skills add obra/superpowers -y [-g] --agent <name>` mapping each of ~40 platform ids to a skills-CLI agent name (SKILLS_AGENT_MAP); platforms the skills CLI doesn't support (lingma/zcode/mimocode) are staged via the claude-code agent into a temp dir and the resulting `.claude/skills` copied into the target platform dir. Both are pure child_process orchestration — no runtime coupling.
  - files: `domains/integrations/openspec.ts`, `domains/integrations/superpowers.ts`
- **OpenSpec-artifact ↔ Superpowers-doc linkage via a hash-verified handoff package** — At design entry, `comet handoff <name> design --write` (script-only; agent handwritten summaries prohibited) reads the OpenSpec open-phase artifacts (proposal.md, design.md, tasks.md, specs/*/spec.md) and emits a compact traceable excerpt: `.comet/handoff/design-context.{json,md}` with source path, line range, sha256, deterministic excerpts (marks [TRUNCATED] when over budget). It writes `handoff_context` + `handoff_hash` into `.comet.yaml`. The Superpowers Design Doc carries minimal frontmatter linking back: `comet_change`, `role: technical-design`, `canonical_spec: openspec`, stored under `docs/superpowers/specs/`. The plan links forward with frontmatter `change`, `design-doc`, `base-ref` (git HEAD before impl). At verify, Comet compares recorded `handoff_hash` vs current to decide whether OpenSpec artifacts changed and skip re-reading tasks.md. OpenSpec delta specs stay canonical — the Design Doc must not create a second requirements spec; it may only write back 'Spec Patches' (add acceptance scenarios / fix ambiguity / add boundaries) to specs/*/spec.md.
  - files: `assets/skills/comet-design/SKILL.md`, `assets/skills/comet-build/SKILL.md`, `assets/skills/comet/reference/comet-yaml-fields.md`
- **Classic five-phase state machine + spec-projection (beta) mode** — 'Comet Classic' is the deterministic runtime (domains/comet-classic/*, bundled to comet-runtime.mjs) that owns the .comet.yaml state machine: `classic-transitions.ts` defines one transition table shared by `comet state transition`, `comet guard --apply`, and archive; `classic-state-command.ts` gates settable vs machine-owned fields; `classic-validate-command.ts` schema-validates. 0.4.0 split machine/run state into `.comet/run-state.json` + append-only `.comet/state-events.jsonl` audit log, with auto-migration of old embedded fields. The handoff has two spec modes selected by `context_compression` (snapshotted from `.comet/config.yaml`): default `off` = the excerpt handoff above; `beta` = a structured 'spec projection' (`spec-context.{json,md}`) that verbatim-projects delta spec files + references supporting artifacts by hash to cut OpenSpec token load without replacing the canonical spec (guard validates spec-context.json structure + source refs). Presets: `full` (5 phases), `hotfix`/`tweak` (skip brainstorming+full plan, open→build→verify→archive). tweak treats delta spec as a first-class artifact and drives build via OpenSpec `apply`, whereas full drives build via Superpowers writing-plans+execution skills.
  - files: `domains/comet-classic/classic-transitions.ts`, `domains/comet-classic/classic-state.ts`, `domains/comet-classic/classic-cli.ts`, `docs/architecture/ARCHITECTURE.md`, `assets/skills/comet-tweak/SKILL.md`
- **TDD mode (tdd|direct) — script-enforced** — `review_mode`/`tdd_mode`/`isolation`/`build_mode` are chosen in one user decision at build phase and are hard constraints enforced by both `comet guard build --apply` and `comet state transition build-complete` (build cannot exit until set). tdd_mode=tdd: under executing-plans the main session loads Superpowers `test-driven-development` once before the first task and follows RED→GREEN→REFACTOR per task; under subagent-driven-development EVERY background implementer/fix agent must load the TDD skill and return explicit RED (failing command+summary) and GREEN evidence — missing either blocks entry into review. tdd_mode=direct: no enforced TDD (default for hotfix/tweak; full workflow may only use build_mode=direct with an explicit `direct_override: true`).
  - files: `assets/skills/comet-build/SKILL.md`, `assets/skills/comet/reference/subagent-dispatch.md`
- **Review mode (off|standard|thorough) — the two review gates** — review_mode governs code-review dispatch and is script-gated at build exit. Build-phase budget (subagent-dispatch.md table, the ONLY allowed count): off = 0 reviewers; standard = per-task reviewer for RISK tasks only (risk signals: cross-module, security surface, concurrency, schema/data migration, public API change, DONE_WITH_CONCERNS, diff>200 lines) + 1 final lightweight review; thorough = per-task reviewer on every task (spec+quality) + 1 final complete review. Fix-round budgets: standard max 1, thorough max 2, then BLOCKED→pause. Under executing-plans there are no per-task subagent reviewers; instead standard=1 final review, thorough=segmented review every 3 tasks + final. CRITICAL findings must be fixed before verify; non-CRITICAL may be accepted with recorded rationale. All via Superpowers `requesting-code-review`; if unavailable, skip but record `<!-- review skipped: skill unavailable -->`. Gate 2 is the verify phase's own review (see verification capability). Note: Comet has NO distinct 'spec review before code' gate — spec quality is guarded only by the open-phase user confirmation + design brainstorming confirmation blocking points.
  - files: `assets/skills/comet/reference/subagent-dispatch.md`, `assets/skills/comet-build/SKILL.md`, `assets/skills/comet-verify/SKILL.md`
- **Verification-before-completion + separated build/verify evidence** — comet-verify step 2 mandatorily loads Superpowers `verification-before-completion`, then branches on `verify_mode` (set by `comet state scale`: full if tasks>3 OR delta-spec capabilities>1 OR changed files>8, overridable). Light = 6 checks (all tasks [x]; diff matches tasks; build passes; tests pass; no obvious security issues; review per review_mode). Full = additionally loads OpenSpec `openspec-verify-change` for spec-scenario coverage, design.md/Design-Doc consistency, proposal goal satisfaction, and delta-spec↔design-doc drift (a user-decision blocking point with options: annotate 'Implementation Divergence', rewind to build via verify-fail, or accept as superseded-by-main-spec). Build and verify evidence are separate and non-substitutable — `comet state record-check <build|verify> --command ... --exit-code` records command text only (Comet NEVER executes it); COMET_SKIP_BUILD=1 is a legacy bypass, not auditable evidence. On failure: a blocking decision (fix vs accept deviation), CRITICAL/IMPORTANT must be fixed, and a hard retry limit (after 3 verify-fail cycles the 4th pauses for explicit user choice).
  - files: `assets/skills/comet-verify/SKILL.md`, `domains/comet-classic/classic-evidence.ts`
- **Reliable nested-skill triggering (three enforcement layers + deterministic routing)** — Comet does NOT call OpenSpec/Superpowers programmatically — it emits instructions for the interactive agent to load them, made reliable by: (1) a standardized imperative phrasing convention 'Immediately execute: Use the Skill tool to load the <skill> skill. Skipping this step is prohibited.' (enforced repo-wide per CLAUDE.md 'Skill 触发表述规范'), always followed by an 'if the skill is unavailable, STOP and prompt to install — do not substitute with normal conversation' clause. (2) An anti-drift Rule (`rules/comet-phase-guard.en.md`) injected every round listing which skills are mandatory per phase and forbidding source writes that skip a phase. (3) A PreToolUse hook `comet-hook-guard.mjs` that HARD-blocks source-code writes when the phase/self-consistency check fails (e.g. phase:build + full + empty design_doc = skipped-design block), with an allowlist for openspec/*, docs/superpowers/*, .claude/*, .comet/*. Phase-to-phase handoff is deterministic, not hardcoded in prose: after `comet guard <phase> --apply` (which must print ALL CHECKS PASSED) the agent runs `comet state next` → `NEXT: auto` (load the SKILL named) | `manual` (respect auto_transition:false, tell user to run it) | `done`. Entry routing uses `comet-intent.mjs route` (CometIntentFrame scorer → hotfix/tweak/full/resume/ask_user/out_of_scope) and read-only `comet-resume-probe.mjs` for ambient resume.
  - files: `assets/skills/comet/rules/comet-phase-guard.en.md`, `assets/skills/comet/reference/scripts.md`, `assets/skills/comet/reference/auto-transition.md`, `domains/comet-classic/classic-hook-guard.ts`, `assets/skills/comet/SKILL.md`

**Strengths**

- Clean separation of concerns: OpenSpec = WHAT (durable spec record + archive), Superpowers = HOW (design/plan/execute/review/verify), Comet = the resumable state machine and gate enforcer that never modifies either vendored toolkit (CLAUDE.md explicitly forbids editing OpenSpec/Superpowers source skills).
- Artifact linkage is hash-anchored and machine-generated, not agent-summarized: design-context.md/json carry source path + line range + sha256; Design Doc frontmatter (comet_change/role/canonical_spec) and plan frontmatter (change/design-doc/base-ref) create bidirectional traceability; verify re-checks handoff_hash to detect spec drift cheaply.
- Gates are script-enforced, not honor-system: guard --apply and state transitions HARD-fail if isolation/build_mode/tdd_mode/review_mode/subagent_dispatch are unset or if evidence is missing; a PreToolUse hook physically blocks phase-skipping source writes; phase jumps via raw `set phase` are hard-blocked (only preset-escalate can legally rewind).
- Deterministic next-step routing (`comet state next` → auto/manual/done) decouples 'phase advancement' from 'auto-invoking the next skill' via auto_transition, so the same skills work in fully-automatic and human-in-the-loop modes.
- Review budget is explicitly bounded and de-duplicated: the subagent-dispatch table is the single source of truth for how many reviews fire; build-phase and verify-phase reviews are dedup'd so the same unchanged diff isn't re-reviewed; model-selection is mandatory per dispatch to avoid silently using the most expensive model.
- Strong context-compaction resilience: brainstorm-summary.md, subagent-progress.md, run-state.json, and state-events.jsonl are durable recovery checkpoints; every sub-skill re-runs phase detection on resume rather than trusting conversation history.
- Node-only, zero-runtime-dependency-on-bash design (0.4.0) makes the whole engine cross-platform (Windows native) and the .mjs launchers are thin wrappers over a compiled TypeScript runtime.

**Limitations**

- The two integration files (openspec.ts/superpowers.ts) are INSTALL-only orchestration — they shell out to `npm i -g @fission-ai/openspec` and `npx skills add obra/superpowers`. Comet is a hard downstream dependent of both external projects' CLI/skill contracts; e.g. it already carries a `--profile` unknown-option fallback and a hack to migrate openspec's hardcoded opencode skillsDir, showing the coupling is brittle to upstream changes.
- There is no distinct 'spec review before code' gate analogous to a formal Gate-1. Spec quality relies on open-phase user-confirmation blocking points and the design brainstorming confirmation; the only fresh-context reviewer dispatch is CODE review (build + verify), not spec review.
- Reliability of nested-skill triggering ultimately depends on the host agent honoring imperative markdown ('Skipping this step is prohibited') plus the injected Rule; only the PreToolUse hook is a true hard stop, and it only exists on hook-capable platforms (Claude Code) — on other platforms enforcement degrades to the soft Rule + guard evidence checks.
- The whole design assumes OpenSpec is initialized in-project (`openspec init`, a terminal command) and Superpowers skills are installed; if either is missing the workflow stops and prompts rather than degrading gracefully.
- Heavy prompt/skill surface area: seven bilingual sub-skills (assets/skills + assets/skills-zh must stay in sync) plus references and a repeated bootstrap boilerplate ('v3') duplicated across every SKILL.md — high maintenance cost and drift risk, partially mitigated by architecture linters and script-freshness tests.
- subagent-driven-development mode requires the platform to have real background-agent dispatch; if it can't be confirmed, Comet must fall back to executing-plans, so the strongest per-task review model isn't universally available.

**Notable for pm**

- This is a direct architectural analog to the user's own pm/OpenSpec/Superpowers stack — Comet is essentially a productized, multi-platform, script-enforced version of the '/comet' idea sitting above the same two toolkits. Worth studying as prior art for the pm conductor's lane routing and gate model.
- Comet's INSTRUCTION-vs-INTEGRATION boundary matches pm's own architectural law almost exactly for the runtime engine (the .comet.yaml state machine never calls a tracker), BUT Comet's integration/*.ts files DO make network/child-process calls to install OpenSpec/Superpowers — that would violate pm's 'engine never opens a network connection' rule. The distinction Comet draws is: install-time provisioning (network OK) vs runtime orchestration (instruction-only). If pm ever adds install helpers, this is the line to draw.
- Concrete linkage mechanism pm could borrow: a script-generated, sha256-anchored handoff package (design-context.json/md) plus reciprocal frontmatter (comet_change/canonical_spec on the design doc; change/design-doc/base-ref on the plan) to keep OpenSpec change ids and Superpowers docs provably in sync — Comet's rule 'OpenSpec is canonical, the design doc may only write back Spec Patches, never a second spec' cleanly resolves the two-source-of-truth problem.
- review_mode off|standard|thorough with an explicit, bounded per-mode review-count table (single source of truth, dedup'd across build+verify) is a cleaner model than ad-hoc 'Gate 1/Gate 2' prose — it makes 'how many reviews and when' deterministic and testable.
- Comet hard-separates build evidence from verify evidence (record-check --command records text but NEVER executes it; COMET_SKIP_BUILD is explicitly non-auditable) — a strong pattern for the pm engine's own 'done requires evidence' stance.
- Comet depends on `@fission-ai/openspec` (note: NOT the same publisher as whatever the user's environment resolves 'openspec' to) installed globally, and on `obra/superpowers` — the exact package coordinates a PM should track for supply-chain/version-compat decisions.
- The state machine is now 3-file (.comet.yaml user projection + .comet/run-state.json machine state + .comet/state-events.jsonl append-only audit) with auto-migration — a maturity step beyond pm's single state.json worth considering if pm state grows machine-owned fields.

### cli+distribution+skill-platform

Comet (npm package @rpamis/comet, v0.4.0-beta.4) is a zero-config CLI "Agent Skill Harness" that installs a five-phase OpenSpec+Superpowers development workflow into 33 different AI coding platforms. It is fundamentally a cross-platform skill installer/transpiler: one set of markdown "Skills" in assets/ is fanned out (copy or symlink) into each platform's native config directory, with rules and PreToolUse hooks translated into 7 different per-platform formats. Beyond installation, it ships a second-order distribution system — /comet-any (the "Skill Creator") lets users compose their own workflow "Bundles" from local skills, which are then eval-gated, review-gated, and distributed to the same 33 platforms via `comet publish`/`comet bundle distribute`. The CLI is built on commander.js; the install engine lives in platform/install/ and domains/skill/; the Bundle/Factory composition system lives in domains/bundle/ and domains/factory/. It bundles i18n (English + Simplified Chinese) across both the CLI UI and the installed skill/artifact content.

**Capabilities**

**Strengths**

- Genuinely broad cross-platform reach: one skill authoring model fans out to 33 AI coding tools, with real per-platform adaptation (different skills dirs, global vs project dirs, rule formats, and 7 hook config schemas) rather than a lowest-common-denominator dump.
- Strong install-safety engineering: atomic tmp+rename writes for the project registry, idempotent hook merging that also cleans up legacy formats, symlink mode that refuses to clobber unmanaged directories, per-file failure counts surfaced in the summary (no silent half-installs), and a postinstall that can never break npm install.
- Coherent two-tier distribution story: (tier 1) the Comet workflow itself distributed as an npm CLI, and (tier 2) user-authored Bundles distributed to the same 33 platforms via publish/distribute — with hash-bound state machine, eval + human-review gates, executable-disclosure confirmations, and a mandatory install --preview before writing files.
- Thoughtful i18n: separates CLI-UI language from installed-artifact language, ships full parallel en/zh skill trees and paired rule variants, and threads the resolved artifact language into downstream OpenSpec/Superpowers prompts.
- Clean architectural layering (app/cli → app/commands → domains/{skill,bundle,factory,integrations} → platform/{install,fs,version}) enforced by a lint:architecture check, mirroring OpenSpec's config conventions for interoperability.

**Limitations**

- Very high surface-area complexity for a beta (0.4.0-beta.4): the CLI exposes ~40 commands/subcommands including two overlapping advanced backends (`creator`/`publish` vs `bundle`) that both serve /comet-any — the layering is powerful but the mental model is heavy and the 'publish/distribute' verbs are easy to confuse with npm publishing.
- Broad platform support is asserted (33 platforms) but adaptation depth is uneven: many platforms only get skills copied with no rules or hooks (e.g. junie, crush, costrict, bob, forgecode, kimicode, antigravity), and several are 'reuse another platform's integration then migrate files' (zcode/mimocode/trae-cn), which is brittle to upstream changes.
- Correctness of the fan-out depends on external CLIs and cache-sniffing: OpenSpec install shells out to `openspec`/npm -g and Superpowers detection pokes at ~/.claude, ~/.codex, ~/.config/opencode plugin caches by hardcoded path shape — fragile if those tools change their cache layout.
- I did not run tests or the build, so 'installs correctly into all 33 platforms' is a claim from code structure and README, not verified behavior; the eval/ Python harness and dashboard were out of scope and unexamined.
- Skill enforcement (phase guards) leans on a single PreToolUse hook translated into each platform's schema — platforms without hook support (a majority of the 33) get advisory rules only, so the 'brainstorming cannot be skipped' guarantee is not uniformly enforced.

**Notable for pm**

- Direct strategic contrast to Claude Code's plugin model: Claude Code plugins are a single-host marketplace (plugin.json + commands/skills/hooks/agents installed via /plugin from a marketplace repo, Claude-Code-only). Comet instead ships as a standalone npm CLI that TRANSPILES the same skill content into 33 platforms' native config dirs, translating rules into 3 formats and hooks into 7. It is a cross-platform installer/compiler, not a plugin registry — the opposite bet from cfdude-plugins' Claude-Code-native marketplace approach.
- The `pm` plugin could borrow Comet's project-registry pattern (~/.comet/installations.json with atomic writes + canonical-path keying) to power cross-repo operations like 'update/upgrade all conductor-managed repos' — directly relevant to /pm:upgrade's per-repo dogfooding sequence.
- Comet's /comet-any 'author a workflow skill, eval-gate it, review-gate it, then distribute' pipeline is conceptually parallel to the pm/OpenSpec propose→gate→apply→archive lifecycle, but productized as a user-facing skill-creation flow with a hash-bound state machine and mandatory install-preview — a possible reference design for making the conductor's lane workflow self-distributable.
- Distribution positioning is worth noting competitively: Comet explicitly supports the generic `npx skills add` channel in addition to its own installer, hedging across the emerging 'skills' packaging ecosystem — a portability stance a Claude-Code-only marketplace does not have.
- Install/uninstall are scope-aware (project vs global) and registry-driven with --all-projects fan-out; if pm ever ships as an installable artifact rather than a marketplace plugin, this is a proven template for multi-repo lifecycle management.

### eval+dashboard

Comet's eval platform is a Python/pytest + Docker harness (uv-managed, under eval/) that quantitatively scores how well a Claude Code "Skill" or the Comet workflow executes. It runs a subject agent inside an isolated Docker workspace, scores the resulting artifacts with a multi-dimension rubric (plus optional LLM-as-judge), and aggregates repeated runs into pass@k (capability ceiling) / pass^k (reliability floor) metrics with a raw-vs-analysis-set denoising discipline. It supports a two-agent "auto_user" interaction mode (subject agent + a user-simulator agent driving multi-turn decision points) and optional LangSmith reporting/trajectory tracing. The "comet dashboard" is a separate, unrelated read-only web app (Node HTTP server in domains/dashboard/*.ts serving a React+Tailwind SPA) that visualizes the live state of a repo's openspec/changes — phase, tasks, verify result, grouped artifacts, risks, and a recommended next command. Context compression is a Comet workflow (not eval) feature that trims Design→Build handoff tokens ~25-30%. Relative to Rob's `pm` conductor plugin, essentially all of this is unique: PM is a zero-dependency Node instruction-layer that never runs code, has no scoring/metrics/Docker/LLM-judge and no web UI. The only conceptual overlap is that both orbit OpenSpec workflow state (phases, tasks, artifacts) and both are "meta" tooling above a spec-driven build lane.

**Capabilities**

- **Rubric scoring (multi-profile, dimension-weighted)** — Three profiles select rubric sets: generic (7 dims: completion, skill_invocation, artifact_presence, instruction_following, interaction_compliance, efficiency, safety_boundary), comet-workflow (9 dims incl. main_flow, gate_guard, artifact_quality, spec_drift, decision_point_compliance, recovery_resilience), authoring-skill (11 dims). Each dimension is a set of binary checks; dimension score = passed/total; final weighted_score is a weighted average. N/A dims (e.g. no required_skills configured) are excluded from weighting to avoid inflated middle scores.
  - files: `eval/scaffold/python/profiles.py`, `eval/scaffold/python/validation/generic_rubric.py`, `eval/scaffold/python/validation/rubric.py`, `eval/scaffold/python/validation/authoring_rubric.py`, `eval/README.md`
- **pass@k / pass^k reliability metrics** — pass@k uses the HumanEval unbiased estimator 1 - C(n-c,k)/C(n,k) (capability ceiling: at least one of k tries succeeds). pass^k = 1.0 iff all n runs passed else 0.0 (reliability floor: every try succeeds). The gap pass@k - pass^k is surfaced as an instability signal for a workflow skill users run repeatedly. Computed per treatment across --count repeated runs.
  - files: `eval/scaffold/python/pass_at_k.py`, `eval/scaffold/python/sample_quality.py`
- **LLM-as-judge (optional, isolated judge)** — BENCH_LLM_JUDGE=1 enables qualitative scoring of workspace artifacts (task_completion, output_quality, instruction_adherence + custom rubric_criteria dims). Judge MUST use a separate config (BENCH_JUDGE_MODEL required; optional BENCH_JUDGE_BASE_URL/AUTH_TOKEN/API_KEY) so the same model can't judge itself. Prefers a direct Anthropic /v1/messages HTTP call to a dedicated judge endpoint; falls back to the host `claude` CLI with inherited ANTHROPIC_* provider env scrubbed. Judge output is additive [RUBRIC-JUDGE], never replaces rule-based scores.
  - files: `eval/scaffold/python/llm_judge.py`, `eval/scaffold/python/generic_llm_judge.py`, `eval/scaffold/python/judge_config.py`
- **Dual-agent (subject + user-simulator) interaction** — auto_user mode drives multi-turn workflows: run-claude-loop.sh starts the subject `claude -p` with the task prompt (stream-json), detects decision points via heuristics (question marks, confirm/choose/proceed keywords), then runs a one-shot user-simulator `claude -p` that approves plans / picks defaults / never refuses, feeding the reply back via --resume so session state persists across turns. Simulator prompt is configurable (simulator-instruction.md / per-profile COMET_SIMULATOR_PROMPT).
  - files: `eval/scaffold/shell/run-claude-loop.sh`, `eval/simulator-instruction.md`, `eval/scaffold/python/profiles.py`
- **Dockerized isolated experiment workspace** — Each experiment creates a host temp dir mounted at container /workspace; the selected treatment's full Skill packages are copied into .claude/skills, a task Dockerfile provides runtime+test deps, validation scripts run inside the container and write _test_results.json. Treatments (control.yaml, comet_full_039/040_beta.yaml) declare which Skills to inject; the scaffold shell (/opt/scaffold-shell) is read-only mounted for the dual-agent loop.
  - files: `eval/scaffold/python/validation/docker.py`, `eval/scaffold/python/validation/runner.py`, `eval/scaffold/shell/docker.sh`, `eval/scaffold/shell/setup.sh`, `eval/local/treatments/comet/comet_full_040_beta.yaml`
- **LangSmith integration (reporting + trajectory tracing)** — An optional langsmith/ suite reuses local tasks/treatments/skills via env path overrides. Each (task,treatment) run becomes a dataset example + experiment; runs report outputs (turns, tool calls, tokens, cost, invoked skills) and per-dimension feedback + checks_pass_rate, so CONTROL vs skill-injected treatments compare side-by-side. Full Claude Code trajectory tracing is delegated to the official langsmith-tracing CC plugin (built once into eval/.cache and mounted at /opt/langsmith-cc-plugin); LANGSMITH_* config auto-derives the TRACE_TO_LANGSMITH / CC_LANGSMITH_* plugin vars. Validation checks self-nest under host-side eval spans via distributed tracing headers.
  - files: `eval/langsmith/README.md`, `eval/scaffold/python/validation/tracing.py`, `eval/scaffold/python/validation/runner.py`, `eval/langsmith/tests/conftest.py`
- **Denoised reporting (raw vs analysis set) + HTML/paper charts** — Reports keep a raw set (all runs, for audit) and an analysis set (headline metrics/pass@k/cost/verdict). Runs are classified included / flagged (suspicious harness/task noise, still counted but called out) / excluded (env noise: API timeout, rate limit, auth/network, Docker/container fail, outer timeout — dropped from headline). Emits 'Insufficient clean data' / 'Inconclusive due to data quality' instead of a misleading verdict. Markdown default; HTML + paper-style figures opt-in via COMET_EVAL_REPORT_CONFIG / --report-config.
  - files: `eval/scaffold/python/report_outputs.py`, `eval/scaffold/python/paper_charts.py`, `eval/scaffold/python/sample_quality.py`, `eval/scaffold/python/evidence.py`, `eval/scaffold/python/attribution.py`
- **comet dashboard (read-only workflow visualizer)** — A Node http server (domains/dashboard/server.ts, default port 4321, auto-increments if busy, 127.0.0.1 only) serves GET /api/dashboard (a freshly collected snapshot per hit, no-store) plus the built React SPA. The collector sweeps openspec/changes (active + archive/), reading .comet.yaml for phase, tasks.md progress, verify result, delta specs, and Comet/Superpowers artifact path-pointers; it groups artifacts by source (OpenSpec / Superpowers / Comet) with generated/not-generated/not-applicable states, computes per-change + project risks, and recommends the next /comet-* slash command. Frontend (main.jsx, ~1250 lines, Vite, theme-aware, 30s auto-refresh, ?demo mode) renders phase stepper, artifact groups, risks, git status.
  - files: `domains/dashboard/server.ts`, `domains/dashboard/collector.ts`, `domains/dashboard/types.ts`, `domains/dashboard/next-action.ts`, `domains/dashboard/risk.ts`, `domains/dashboard/web/src/main.jsx`, `domains/dashboard/README.md`
- **Context compression (Design→Build handoff, Beta)** — NOT part of eval — a Comet workflow token optimization. comet-handoff.mjs reads context_compression in .comet.yaml: off embeds full spec excerpt in the handoff JSON; beta keeps only the Design Doc + a SHA256 hash reference to the spec, cutting Build-phase input tokens. Handoff written to openspec/changes/<name>/.comet/handoff/design-context.json with hash recorded back to .comet.yaml. Dry-run benchmarks report ~25-33% total-token savings scaling with spec size, 100% test pass rate, ~95% spec coverage (minor edge-requirement loss).
  - files: `docs/operations/CONTEXT-COMPRESSION.md`, `scripts/context-execution-benchmark.mjs`
- **Task/treatment corpus + eval-as-publish-evidence** — ~20 tasks under eval/local/tasks (each = task.toml + instruction.md + environment/Dockerfile + validation/test_*.py, registered in index.yaml); categories generic vs comet. The user-facing entry is `comet eval <target> --collect|--html|--quick`: --collect is a cheap discovery precheck, --html runs the real eval and produces browsable reports, results become publish-readiness evidence for /comet-any (hash-matched preferenceHash + eval evidence gate publishing). eval never publishes.
  - files: `eval/local/tasks/index.yaml`, `eval/scaffold/python/tasks.py`, `eval/scaffold/python/treatments.py`, `eval/scaffold/python/manifests.py`, `docs/operations/EVAL-USAGE.md`

**Strengths**

- Statistically honest metrics: separates capability (pass@k, HumanEval unbiased estimator) from reliability (pass^k) and surfaces the instability gap — the right framing for a workflow tool users run repeatedly, not a single pass rate.
- Denoising discipline (included/flagged/excluded, raw vs analysis set) plus explicit 'Insufficient clean data' / 'Inconclusive' verdicts prevents environment noise (timeouts, rate limits, Docker failures) from producing misleading win/loss conclusions.
- Judge isolation is carefully engineered: a separate judge model/endpoint is mandatory, host ANTHROPIC_* provider env is scrubbed in the CLI-fallback subprocess, and judge output is additive rather than replacing rule-based scores — avoids the model grading itself.
- Dimension-weighted rubrics with N/A handling (unconfigured dims excluded from weighting) avoid inflated middle scores, and skills_invoked is derived only from real Claude Code tool-call/skill-load events, not reverse-engineered from artifacts — hard to game.
- Strong isolation and reproducibility: per-experiment Docker workspace, full Skill-package injection via named treatments (039 vs 040-beta snapshots), dry-run benchmark modes that need no API tokens.
- Dashboard is deliberately read-only and defensive: per-change errors are swallowed so one malformed .comet.yaml can't blank the page, missing files map to empty states, path-traversal defense on static serving, 127.0.0.1 bind, port auto-retry — safe to point at any repo.
- Clean separation: eval provides publish evidence but never publishes; dashboard observes but never mutates; both compose over the same OpenSpec/Comet workflow model.

**Limitations**

- pass^k as implemented is a coarse lower bound: 1.0 only if literally every observed run passed, else 0.0 — no graded reliability estimate; a 9/10 run set scores identically to 0/10 on pass^k.
- Dual-agent decision-point detection is heuristic (grep for '?', confirm/choose/proceed keywords), so it can misclassify a subject turn (drive a simulated reply into a non-question, or miss a decision phrased without those cues); the loop also relies on brittle stream-json parsing and completion-keyword matching to decide when the workflow is done.
- Heavy external dependency surface for full runs: Docker, the `claude` CLI, Anthropic API tokens/proxy config, and (optionally) LangSmith — a lot must be wired correctly, and much of the failure taxonomy exists precisely because these are flaky.
- Docs and much of the UI are Chinese-first (eval/README.md, dashboard, next-action/risk suggestion strings, CONTEXT-COMPRESSION.md), which narrows accessibility for non-Chinese-reading operators.
- Context-compression benchmark numbers are dry-run / simulated-model results (test date 2026-06-07, repeat=1), not live-API measured — the ~25-33% savings and 95% spec-coverage figures are modeled, and beta mode carries an acknowledged small spec-coverage loss.
- Dashboard is a point-in-time snapshot (fresh collect per request + 30s client poll) with no history/trend view — it shows current workflow state, not evaluation results or metrics over time (eval and dashboard are entirely separate systems that don't feed each other).
- LangSmith trajectory tracing depends on building an external third-party plugin from GitHub into a cache dir, a live network/build dependency at first run.

**Notable for pm**

- Near-zero overlap with the `pm` conductor plugin: PM is a zero-dependency Node instruction-layer that emits CLAUDE.md rules/briefings and never runs code, opens a network connection, spawns Docker, or scores anything. Comet's eval is a Python/uv/pytest/Docker measurement harness with LLM judges and API calls — the opposite architectural posture. This is confirmation of differentiation, not a feature to copy into PM.
- The one genuine conceptual adjacency is the dashboard: it visualizes openspec/changes phase + tasks + verify + artifacts + a recommended next command — the same workflow-state surface PM's conductor tracks as epics/status in .conductor/state.json and renders as generated PROJECT.md. Comet does it as a live read-only web SPA; PM does it as agent-facing markdown.
