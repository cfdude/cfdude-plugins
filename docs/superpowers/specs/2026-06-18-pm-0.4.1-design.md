# Design: pm 0.4.1 — robust, self-guiding upgrade

- **Date:** 2026-06-18
- **Plugin:** `pm` (in `cfdude/cfdude-plugins`)
- **Version:** 0.4.0 → **0.4.1** (patch: engine behavior + docs; no schema change)
- **Engine:** `plugins/pm/scripts/conductor.mjs` (single file, zero deps, Node 18+)
- **Tests:** `plugins/pm/scripts/conductor.test.mjs` (subprocess integration, `node:test`)

## Problem

The `/pm:upgrade` flow has an implicit, undocumented ordering dependency, and running it out
of order fails silently.

After a plugin version bump, Claude Code does not refresh a running session's command text,
hooks, or `${CLAUDE_PLUGIN_ROOT}` until the user runs `/reload-plugins` or restarts. So if a
user runs `/pm:upgrade` before that reload takes effect, `${CLAUDE_PLUGIN_ROOT}` still points
at the **old** version's cache dir, the **old** engine runs, and its `pluginVersion()` (which
reads its own dir's `plugin.json`) returns the old version — so `upgrade` **silently
re-stamps `pmVersion` to the old version** and reports success. Observed live in the
`personal-finance` repo (0.3.0 → 0.4.0): the old engine would have re-stamped 0.3.0.

Two gaps:
1. **No safety:** running a stale engine silently does the wrong thing instead of refusing.
2. **No guidance:** the required sequence (update plugin → reload/restart → `/pm:upgrade`) is
   nowhere documented, and nothing tells the user when a reload is needed.

### Hard boundary (cannot be fixed by the plugin)

A plugin cannot trigger `/reload-plugins` or a restart — those are Claude Code client
actions. The command text, hooks, and `${CLAUDE_PLUGIN_ROOT}` will lag until the user does
that. So the sequence cannot be *eliminated*; the plugin can only stop doing the wrong thing
silently and guide the user precisely.

## Goals / Non-goals

**Goals**
- `upgrade` never silently re-stamps an old version when a newer one is installed — it refuses
  with a clear, actionable message.
- The SessionStart nudge appears as soon as a newer version is installed (even before a
  reload) and states the full sequence.
- The canonical update sequence is documented (README + command doc).

**Non-goals**
- **No auto-upgrade** on session start (rejected during brainstorming — auto-rewriting
  `state.json`/`CLAUDE.md` without explicit intent breaks the conductor's dormant ethos, and
  doesn't remove the reload anyway).
- **No command-resolves-newest / re-exec** (adds complexity and can't escape the reload for
  command/hook freshness; the staleness guard covers safety).
- No schema change; no data migration.

## Decisions (locked during brainstorming)

1. Approach = **Robust + guided** (staleness guard + smarter nudge + docs). Not auto, not docs-only.
2. The cache-scan root is **env-overridable** (`PM_CACHE_ROOT`) for testability.

---

## Design

### 1. `newestInstalledVersion()` — detection primitive

Add a helper that returns the highest pm version present in the plugin cache (or `null` if it
can't tell):

- Cache root resolution: `process.env.PM_CACHE_ROOT` if set, else
  `path.join(os.homedir(), ".claude", "plugins", "cache")`. (Add `import os from "node:os"`.)
- Scan `<cacheRoot>/<marketplace>/pm/<version>/.claude-plugin/plugin.json` (two directory
  levels: any marketplace, the `pm` plugin dir, any version dir). For each readable
  `plugin.json`, collect its `version`.
- Return the max by `cmpVer`, or `null` if none found / cache dir absent (wrapped in
  try/catch like `activeChangeIds()` — never throws). **Per-entry resilience:** each
  `plugin.json` read is individually guarded (skip on missing/malformed) so one bad version dir
  doesn't collapse the whole scan to `null`; only a genuinely absent/unreadable cache root
  yields `null`.

Why env-overridable: tests set `PM_CACHE_ROOT` to a temp fixture containing
`mp/pm/<ver>/.claude-plugin/plugin.json` files to exercise the staleness logic deterministically.

### 2. Staleness guard in `upgrade()`

At the very start of `upgrade()` (after the `isInitialized()` check), before loading/mutating
state:

```
const running = pluginVersion();          // the engine actually executing
const newest  = newestInstalledVersion(); // highest in the cache
if (running && newest && cmpVer(newest, running) > 0) {
  // a strictly newer pm is installed than the engine running this command
  process.stderr.write(
    `conductor: this is pm ${running}, but ${newest} is installed — your session is still ` +
    `running the old engine.\n` +
    `Run /reload-plugins (or restart Claude Code), then /pm:upgrade again.\n` +
    `(Running the engine directly from a checkout? Set PM_CACHE_ROOT to override.)\n`);
  process.exit(1);
}
```

- Exit **non-zero**, mutate **nothing** (no `saveState`, no `writeRules`, no `render`). This
  replaces the current silent re-stamp-to-old-version behavior.
- When `running === newest` (normal post-reload state) or `newest` is `null` (single version /
  can't tell), the guard is a no-op and `upgrade` proceeds exactly as in 0.4.0.

### 3. Smarter SessionStart nudge in `brief()`

Today the nudge compares the stamped `pmVersion` against the **running** engine's version, so
in the post-update/pre-reload window (stale engine running, stamped == stale) it may not fire
at all. Change the comparison to use `newestInstalledVersion()` (filesystem-based, independent
of which engine runs the hook):

**This REPLACES the existing nudge block** (the current `running = pluginVersion()` /
`cmpVer(stamped, running) < 0` push) — it is rewritten in place, not added alongside, so the
brief emits **at most one** upgrade nudge. Logic:

- `stamped = state.pmVersion || "0.0.0"`, `newest = newestInstalledVersion()`.
- **If `newest !== null`** (cache scan succeeded): push the nudge iff `cmpVer(stamped, newest) < 0`:
  > ⚠ pm `<stamped>` → `<newest>` available — run `/reload-plugins` (if you just updated the
  > plugin), then `/pm:upgrade`.
  If `cmpVer(stamped, newest) >= 0` (nothing newer than the stamp — current, or stamp ahead of
  any cached version) → **no nudge**. This is the intended behavior, not a gap.
- **If `newest === null`** (cache dir absent/unreadable — unusual layout): fall back to the
  prior behavior exactly — `running = pluginVersion()`, push iff `cmpVer(stamped, running) < 0`
  with the original `<stamped> → <running> since this repo was set up` wording.

So there is exactly one push site per branch; behavior never regresses for an unusual install.
The newest-installed comparison (filesystem-based) is what lets the nudge appear pre-reload.

### 4. Documentation

- **`plugins/pm/README.md`** — new "Updating pm in your projects" section:
  1. Update the plugin (marketplace auto-update, or `/plugin`).
  2. **`/reload-plugins`** (or restart Claude Code) — this is what registers the new command
     text, hooks, and engine into the running session; without it `/pm:upgrade` runs the old
     engine.
  3. **`/pm:upgrade`** in each project that uses pm (idempotent; refreshes the CLAUDE.md rules
     block, stamps the new version). The conductor's SessionStart nudge reminds you per repo.
  Explain *why* step 2 is required (Claude Code loads commands/hooks/engine at session start).
- **`plugins/pm/commands/upgrade.md`** — add a note at the top: if the briefing shows an
  upgrade is available, run `/reload-plugins` first so this command uses the new engine; if you
  see "this is pm <old> but <new> is installed", that's the reload reminder.

### Version / migration

- Bump `plugins/pm/.claude-plugin/plugin.json` and the `pm` entry in
  `.claude-plugin/marketplace.json` to **0.4.1**.
- **No migration entry, no schema change.** The rules block is unchanged. `upgrade`'s
  unconditional `writeRules()` + the nudge still deliver any rules refresh on existing repos.

### Honest limitation (document in CHANGELOG)

The staleness guard ships *inside* 0.4.1, so the very first upgrade *into* 0.4.1 still runs the
old 0.4.0 engine until the user reloads (the guard isn't present yet). From 0.4.1 forward,
every future upgrade is self-guarding. CHANGELOG states this.

## Testing (TDD, `node:test`, subprocess integration)

Run: `node --test plugins/pm/scripts/conductor.test.mjs`. Tests use `PM_CACHE_ROOT` pointed at
a temp fixture cache, and `CLAUDE_PLUGIN_ROOT` to control the *running* engine's reported
version (existing `fixturePluginRoot(version)` helper for the latter).

**REQUIRED prerequisite — isolate `PM_CACHE_ROOT` in the `run()` helper.** The `run()` helper
spreads `...process.env`, so without isolation `newestInstalledVersion()` would scan the
developer's *real* `~/.claude/plugins/cache` (which contains 0.4.0+), making the guard fire and
the nudge mis-fire in the many existing tests that don't set `PM_CACHE_ROOT` — breaking
~4 of them. The helper MUST default `PM_CACHE_ROOT` to an isolated empty temp dir created once
for the suite, overridable per-call:
```js
const EMPTY_CACHE = fs.mkdtempSync(path.join(os.tmpdir(), "pm-empty-cache-"));
export function run(args, { cwd, env = {} } = {}) {
  return execFileSync("node", [ENGINE, ...args], {
    cwd,
    env: { ...process.env, CLAUDE_PROJECT_DIR: cwd, PM_CACHE_ROOT: EMPTY_CACHE, ...env },
    encoding: "utf8",
  });
}
```
With an empty cache, `newestInstalledVersion()` returns `null` for every existing test → the
guard is a no-op and the nudge falls back to the running-version comparison → all prior tests
retain their original semantics. New 0.4.1 tests pass `PM_CACHE_ROOT` explicitly to exercise
the new paths.

Helper: a `fixtureCache(versions[])` that creates `<tmp>/mp/pm/<ver>/.claude-plugin/plugin.json`
for each version and returns the cache root path.

1. **`newestInstalledVersion` picks the max:** a fixture cache with `0.3.0`, `0.4.0`, `0.4.1`
   (and a `0.10.0` to confirm semver-not-string) → the engine treats `0.10.0` as newest.
   (Exercised via the nudge/guard behavior below, since the helper is internal.)
2. **Guard refuses a stale upgrade:** run `upgrade` with `CLAUDE_PLUGIN_ROOT` = a fixture
   reporting `0.4.0` and `PM_CACHE_ROOT` = a cache whose max is `0.4.1` → non-zero exit, message
   names both versions, and `state.json` `pmVersion` is unchanged (no mutation).
3. **Guard is a no-op when current:** `CLAUDE_PLUGIN_ROOT` reports `0.4.1` and cache max is
   `0.4.1` → `upgrade` proceeds, stamps `pmVersion: 0.4.1` (normal behavior).
4. **Guard no-op when cache unreadable:** `PM_CACHE_ROOT` = a nonexistent dir → `upgrade`
   proceeds normally (newest is `null`).
5. **Nudge uses newest-installed:** a repo stamped `0.3.0`, `PM_CACHE_ROOT` max `0.4.1` → the
   brief contains `pm 0.3.0 → 0.4.1 available` and the `/reload-plugins` instruction, even when
   the running engine (via `CLAUDE_PLUGIN_ROOT`) is `0.3.0`.
6. **Regression:** all existing 0.4.0 tests still pass.

## Acceptance criteria

- Running `/pm:upgrade` from a stale session (older engine than the newest installed) refuses
  with a message naming both versions and the reload step, and does not change `state.json`.
- Running `/pm:upgrade` from a current session upgrades normally (idempotent).
- The SessionStart briefing shows an upgrade-available nudge with the full sequence whenever a
  newer pm version is installed than the repo's stamped `pmVersion`.
- README documents the update sequence and why the reload is required.
- Full `node:test` suite green; versions at 0.4.1.

## Risks / open questions

- **Cache layout assumption:** `newestInstalledVersion()` assumes the
  `<cacheRoot>/<mp>/pm/<version>/` layout (verified against the live cache). If Claude Code
  changes that layout, the scan returns `null` and the guard/nudge degrade gracefully to no-op
  / running-version comparison — never throwing, never blocking. Documented as a known
  dependency on the cache layout.
- **Dev checkouts:** an engine run directly from a repo checkout (not the cache) compares its
  own version against the cache max; if the cache happens to hold a newer version, the guard
  could refuse. Acceptable — a developer running the engine directly can set `PM_CACHE_ROOT` or
  reload; normal users always run via the cache.
