# Design: pm 0.4.0 — planned status, stamp-on-change, roadmap on-ramp + cleanups

- **Date:** 2026-06-18
- **Plugin:** `pm` (in `cfdude/cfdude-plugins`)
- **Version:** 0.3.0 → **0.4.0** (minor: additive status value + behavior refinements)
- **Engine:** `plugins/pm/scripts/conductor.mjs` (single file, zero deps, Node 18+)
- **Tests:** `plugins/pm/scripts/conductor.test.mjs` (subprocess integration, `node:test`)

## Background

0.3.0 made the conductor lane-agnostic. Field use across three repos surfaced follow-ups:

1. **PROJECT.md timestamp churn** — `render()` re-stamps `Last rendered` on every render, so a
   read-only `/pm:status` produces a committable diff. The timestamp is wanted, but it should
   move only when the board's content actually changes.
2. **Planned-but-unproposed OpenSpec roadmap is invisible** — an epic intended to become an
   OpenSpec proposal, registered before `opsx:propose`, is `lane: openspec` with no change dir,
   so `missing()` flags it `⚠ no change on disk` and excludes it from NEXT UP. 0.3.0 covered
   *non*-OpenSpec lanes but left this sibling gap: the conductor can't hold a forward OpenSpec
   roadmap as ordered backlog.
3. **Three small cleanups** carried from the 0.3.0 reviews/field use.

## Goals / Non-goals

**Goals**
- The `Last rendered` timestamp moves only when board content changes (no read-only churn).
- A `planned` status lets the conductor hold intended-but-unproposed work as ordered backlog,
  distinct from a genuinely-broken missing change.
- Smooth roadmap → proposed handoff (auto-transition on `sync` when the change dir appears).
- A documented (not auto-parsed) on-ramp for importing an existing roadmap doc.
- Fix the three known cleanups.

**Non-goals**
- **No roadmap auto-import / parsing.** Every project's roadmap is a different markdown shape;
  the only on-ramp is a documented `/pm:epic add` workflow the AI session performs.
- **No new lifecycle machinery** beyond the single `planned` status value.
- Auto-transition is scoped to the **openspec** lane (change-dir appears). Other lanes stay
  manual re-triage in 0.4.0.

## Decisions (locked during brainstorming)

1. Brief surfacing of planned epics = **count + pointer** (`planned: N — see PROJECT.md`);
   NEXT UP stays actionable-only.
2. On `opsx:propose` + `/pm:sync`, a matching planned epic **auto-transitions** `planned → untriaged`.
3. Bundle the three cleanups into this 0.4.0 release.

## Design

### A. PROJECT.md — stamp on content change

`render()` (currently writes `> Last rendered: ${new Date().toISOString()}` unconditionally at
line ~375, then `fs.writeFileSync(PROJECT_MD, …)` at ~438).

New behavior:
1. Build the full markdown `content` exactly as today, including a fresh `Last rendered` line.
2. Read the existing `PROJECT.md` (if any).
3. Strip the `Last rendered` line from **both** new and existing via the regex
   `/^> Last rendered: .*$/m` (replace with a constant sentinel or remove the line).
4. If the existing file exists **and** its stripped form equals the new stripped form → **do
   not write** (preserve the file and its old timestamp). Emit a stderr note like
   `conductor: PROJECT.md unchanged`.
5. Otherwise → write `content` (with the fresh timestamp).

Only line ~375 is time-varying; the "Recent detours" table's `When` column comes from the
append-only `detours.log` (stable event times, not regenerated), so content-equality is reliable.

Result: read-only `/pm:status` with no state change → no rewrite, no diff. A `tasks.md`/plan
checkbox change, a status/priority change, or a detour push/pop → content differs → rewrite +
fresh timestamp. The timestamp becomes an honest "last time the board changed."

### B. `status: planned`

**Known statuses.** Introduce `KNOWN_STATUSES = ["untriaged", "queued", "active", "paused",
"planned", "archived"]`. `add-epic` validates `--status` against this set (validate-on-write).
Reads tolerate unknown values (render the literal) — same pattern as `lane`.

**`missing()` exclusion.** Change:
```
missing(e) = e.lane === "openspec" && !e.present && !isArchived(e.id) && e.status !== "planned"
```
A planned openspec epic with no change dir is intended, not broken: no `⚠ no change on disk`
marker, and not excluded from anything on the missing basis.

**Brief (count + pointer).** NEXT UP filter stays `["queued", "untriaged"]` (actionable). After
the NEXT UP block (or standalone if NEXT UP is empty), add one line when any non-archived
planned epics exist:
```
planned: <N> — see PROJECT.md
```
`N` = count of resolved epics with `status === "planned"` and not archived. Keeps the
compaction-surviving brief lean.

**PROJECT.md Epics table.** Planned epics render like any other row (status shows `planned`),
sorted by priority then lane, no missing marker — so the full ordered roadmap is browsable in
PROJECT.md. (This is the "see PROJECT.md" target.)

**Registering.** `/pm:epic add <id> "<title>" <lane> <priority> --status planned` is the path to
register a roadmap item. `planned` is valid for any lane (intended future work); for the
openspec lane it additionally suppresses the missing warning per the predicate above.

### C. Auto-transition on propose (`sync`)

In `sync()`, before the additive scan: for each existing epic in `state.epics` with
`status === "planned"` whose id is in the on-disk active OpenSpec change set
(`activeChangeIds()`), flip `status` to `"untriaged"` (it's been proposed; now a real epic to
triage). Emit a stderr note per transition (when not quiet). Then proceed with the existing
additive OpenSpec + plan import. Save once.

Scoped to the openspec lane (the change-dir-appears signal). A planned epic in another lane is
not auto-transitioned in 0.4.0.

### D. Cleanups

**D1. `parseFlags` valueless-flag guard.** Today a flag with no following value is assigned the
string `"true"`; a trailing `--id` therefore yields `f.id === "true"`, which passes the id
regex and creates an epic literally named `true`. Fix: `parseFlags` assigns the boolean `true`
(not the string) to a valueless flag. `add-epic` then requires `--id` to be a **string**
matching `^[a-z0-9][a-z0-9._-]*$` (a boolean fails the `typeof`/regex check) and errors
non-zero. `--lane`/`--status` validation already gate non-string/invalid values.

**D2. `find` → portable glob in command-doc fallbacks.** Five command files
(`init.md`, `status.md`, `sync.md`, `epic.md`, `upgrade.md`) use
`ENGINE=$(find ~/.claude -name conductor.mjs -path '*pm*' | head -1)` as the fallback when
`${CLAUDE_PLUGIN_ROOT}` is unset. Replace with a glob that needs neither `find` nor `fd`:
```
ENGINE=$(ls -t ~/.claude/plugins/cache/*/pm/*/scripts/conductor.mjs 2>/dev/null | head -1)
```
`ls -t` returns newest-first, so the latest installed version wins. Portable (ls + shell glob
are universal); satisfies the project's no-`find` preference without adding an `fd` dependency.

**D3. Roadmap on-ramp guidance (docs/skill, no parser).** Add a short "Importing an existing
roadmap" section to `plugins/pm/README.md` and `plugins/pm/skills/conductor/SKILL.md`: if the
project has a roadmap doc (any markdown), in an interactive session read it and register each
item with `/pm:epic add … --status planned` (choosing the execution lane). The `conductor`
skill should proactively offer this when it detects a roadmap-style doc alongside a sparse
conductor. Explicitly: the conductor does **not** parse roadmap files.

### E. Rules block

Update `rulesBlock()` to (1) mention `planned` as a status for intended-but-unproposed work,
and (2) one line on the roadmap on-ramp. The block already lists lanes from 0.3.0; this is an
additive wording change.

### F. Version / migration

- Bump `plugins/pm/.claude-plugin/plugin.json` and the `pm` entry in `.claude-plugin/marketplace.json` to **0.4.0**.
- **No data migration entry required.** `planned` is an additive status value — no existing
  `state.json` data is rewritten or invalidated. The refreshed rules block reaches existing
  repos through `upgrade()`'s **unconditional** `writeRules()` (it runs regardless of migration
  entries), and the SessionStart nudge fires for any repo stamped `< 0.4.0`. Running
  `/pm:upgrade` on a 0.3.0 repo: 0 migrations apply, rules refresh, `pmVersion → 0.4.0`,
  re-render — idempotent. (Stated explicitly to preempt a "missing 0.4.0 migration" finding:
  the migration-discipline rule exists so existing repos receive changes; the unconditional
  `writeRules` + nudge already deliver that here, and there is no data to migrate.)
- New `plugins/pm/CHANGELOG.md` `0.4.0` entry documenting all of the above, including the
  upgrade note ("run `/pm:upgrade` after updating").

## Testing (TDD, `node:test`, subprocess integration)

Run: `node --test plugins/pm/scripts/conductor.test.mjs`.

1. **Stamp-on-change:**
   - `render` twice with no state change → `PROJECT.md` byte-identical across the two runs
     (including the `Last rendered` line).
   - Mutate state (e.g. change an epic's priority via `writeState`), `render` → the content
     differs and the timestamp line advances (assert the body changed; assert a `Last rendered`
     line is present).
2. **planned — not missing, not in NEXT UP, in table, counted in brief:**
   - A `lane: openspec`, `status: planned`, no-change-dir epic renders WITHOUT `no change on
     disk`, does NOT appear in the brief's NEXT UP, and the brief contains `planned: 1 — see
     PROJECT.md`. PROJECT.md's Epics table shows the epic with status `planned`.
3. **add-epic status validation:** `--status planned` accepted; an unknown `--status` rejected
   (non-zero exit).
4. **Auto-transition:** a planned openspec epic whose `openspec/changes/<id>/` dir is then
   created → after `sync`, its status is `untriaged` (and it now appears in NEXT UP).
5. **parseFlags guard:** `add-epic --lane superpowers --id` (trailing, no value) exits non-zero
   and creates no epic; a normal `--id x` still works.
6. **Doc/glob changes** verified by `git grep` (no `find ~/.claude` remains in command docs; the
   `ls -t … cache … conductor.mjs` glob present) — not a `node:test` case.
7. **Regression:** all existing 0.3.0 tests still pass.

## Acceptance criteria

- A repo can register a future-OpenSpec roadmap as `planned` epics that appear as ordered
  backlog in PROJECT.md and as a `planned: N` count in the brief, with NO `no change on disk`
  warnings and no NEXT UP pollution.
- `opsx:propose` + `/pm:sync` flips the matching planned epic to `untriaged` automatically.
- A read-only `/pm:status` produces no `PROJECT.md` diff; a real change does.
- `add-epic` rejects a valueless `--id`.
- Command-doc fallbacks contain no `find`.
- `/pm:upgrade` on a 0.3.0 repo refreshes rules + stamps 0.4.0, idempotently; no data loss.
- Full `node:test` suite green.

## Risks / open questions

- **Auto-transition scope:** only openspec-lane planned epics transition on `sync`. A planned
  superpowers epic whose plan file later appears is not auto-transitioned — acceptable for
  0.4.0; revisit if it bites.
- **Content-equality fragility:** stamp-on-change relies on render output being deterministic
  except for the `Last rendered` line. This holds today (detour `When` times come from the
  append-only log, not regeneration). If a future render adds another wall-clock value, this
  invariant must be preserved (documented here as a constraint).
